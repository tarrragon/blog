# Capability Outsourcing Depth（外包深度）

> **角色**：本卡是 `saas-tech-selection` 的支撐型原則（principle）、被 domain / event 切分（commodity domain check）與技術維度展開（各維度的託管預設）共同引用。
>
> **何時讀**：判斷一塊能力或一個 domain「該自建還是外包、外包到哪個深度」時；尤其是發現一個 vendor 同時涵蓋多個維度（資料庫 + 認證 + 儲存）、不知道該怎麼歸位時。

外包一塊能力不是「自建 / 不自建」的二元選擇、而是有深度的。同樣是「不自己寫」、把維運交出去跟把整塊能力連業務邏輯一起交出去、保留的控制權與遷出代價差一個量級。判斷外包到哪個深度、跟「買還是建」是同一個決策的兩半。

## 三種深度

| 深度                       | 你外包什麼、還擁有什麼                                                    | 典型例子                                                  | 遷出代價 |
| -------------------------- | ------------------------------------------------------------------------- | --------------------------------------------------------- | -------- |
| managed 基礎設施           | 外包維運（備份、修補、failover、擴容）、schema / query / 資料模型仍是你的 | managed 關聯式 DB、managed 快取、managed K8s              | 低–中    |
| feature SaaS（能力即服務） | 外包整塊能力與內部業務邏輯、只消費一組 API                                | 託管身份服務、金流、搜尋服務、Email 投遞、表單 / 問卷平台 | 中–高    |
| 跨能力 bundle              | 一個 vendor 同時交付多塊能力、用整合當賣點                                | 同時提供資料庫 + 認證 + 儲存 + 即時同步的後端平台         | 高       |

managed 基礎設施撞牆時你改得動的邊界很寬 —— 慢查詢自己優化、index 自己加、只有底層硬體與維運在 vendor 手上。feature SaaS 撞到 vendor 沒開放的客製、就只能在它之外再搭一層、邊界縮到 vendor 的擴展點為止。bundle 的遷出代價最高、原因不在任何單一能力、而在多塊能力被同一套整合綁在一起。

## 深度適用在兩個層級

同一條深度軸、用在兩個不同的決策層級：

- **技術維度層**（state-storage、deployment、cache、async-queue、observability…）：多落在 managed 基礎設施深度 —— 你還擁有這個 domain 的資料與模型、只把 runtime / 維運外包。day-one 預設「用 managed」就是這層的判斷。
- **領域層**（domain / event 切分時）：多落在 feature SaaS 深度 —— 買掉的是整個 domain，vendor 連能力、資料模型、UI 一起擁有，這個 domain 根本不進你的 codebase。買金流不是「選一個 managed 金流基礎設施」、是「付款 domain 變成一條整合邊界」。

領域層的外包槓桿比維度層大：維度層省維運、領域層直接刪掉一整塊要建的 domain（連帶刪掉它要展開的技術維度）。

## 跨能力 bundle 打破「逐維度各自選」的假設

技術維度的 day-one 託管預設、隱含「每個維度各挑一個 managed 服務」。bundle 打破這個假設 —— 一個後端平台可以一次涵蓋 state-storage + 認證 + 物件儲存 + 即時推送。判讀單位仍然是逐維度 / 逐 domain：可以只用 bundle 的資料庫當 managed 基礎設施、認證自建或接別的 feature SaaS；也可以反過來。bundle 的價值（同一套身分貫穿多塊能力、省掉接縫）與鎖定（搬走一塊要拆它跟其他塊的整合）同源。發現一個 vendor 同時 cover 多維度時、不要當成「整包接受或整包拒絕」、逐塊問「這塊用它的值不值得換取整包的遷出代價」。

## 整合接縫成本

每多買一塊 feature SaaS、就多一道接縫 —— 資料跨服務重複、跨服務一致性、整合維護。買越多塊、系統的真正複雜度從「每塊能力內部」移到「能力之間的縫」。這是外包換來的隱性帳、跟「省下維運」是同一個決策的反面、評估買 vs 建時要跟省下的成本一起算。決策記錄裡每多一個外部服務、就在規模 tripwire 之外多記一條「這道接縫由誰維護、資料一致性怎麼保證」。

## 怎麼用在訪談

1. 切 domain 時，逐 domain 過 commodity domain check：這塊是不是現成 feature SaaS 已經做完的標準能力（認證、金流、表單 / 問卷、搜尋、通知、後台 CRUD）？命中 → 標為「買、整合邊界」、不進 build scope。
2. 展開技術維度時，維度的 day-one 託管預設標出它的外包深度（managed 基礎設施 / feature SaaS / 折進 bundle）。
3. 發現候選 bundle 時，逐維度判斷哪幾塊用它、把「用 bundle 哪幾塊」與「整包遷出代價」一起記進決策。
4. 買掉的每塊能力，在決策記錄補一條接縫維護與一致性的承擔者。
