# Gap Audit and Backfill（缺卡掃描與回填）

本 reference 為「存量模組的卡片 audit」情境：找出缺卡、排優先序、批次補卡後把文章接上卡、以及跨模組大規模重構的執行順序。

適用：模組級缺卡掃描、回填連結規劃、卡片目錄的工具鏈登記、跨模組重構。
不適用：單張卡的撰寫格式（見 SKILL.md 路由到 card-authoring）。

> **自包含聲明**：閱讀本文件不需要先讀其他 reference。缺卡判定矩陣在本文內完整給出。

---

## 缺卡判定矩陣

術語處理分三級：**行內展開**（正文 1-2 句定義、所有讀者付閱讀成本）、**連卡**（正文用術語連到卡、需要的讀者點進去）、**裸用**（不解釋不連、基線外讀者斷路）。判定用兩軸：

- **軸一、術語在該文的角色**：主線（該文的核心論證繞著它轉）／支撐（論證用到但該文不教）／背景（順帶出現）
- **軸二、相對基線**：基線＝模組讀者定位 ∪ 閱讀路線中前置篇章教過的主線術語

| 術語角色 | 基線外                        | 基線內                     |
| -------- | ----------------------------- | -------------------------- |
| 主線     | 行內展開＋連卡                | 行內展開（該文就是在教它） |
| 支撐     | **連卡（卡不存在 → 先建卡）** | 連卡（預設）或裸用         |
| 背景     | 連卡                          | 可裸用                     |

**「支撐 × 基線外 × 無卡」就是缺卡**——取代「感覺這個詞需要解釋」的直覺判斷。主線術語永遠行內展開（讀者跳出去讀卡、回來論證已斷線）；連卡密度不隨文章深度遞減（空降讀者的回路）；同一術語每篇第一次出現連卡、後續裸用。

## 模組 audit 流程

1. **詞頻掃描**：列該領域的術語候選清單、grep 各術語的出現篇數、按篇數排序：

   ```bash
   for t in "term_a" "term_b" "term_c"; do
     echo "$(grep -l "$t" <module_dir>/*.md | wc -l) $t"
   done | sort -rn
   ```

2. **比對卡片目錄**：逐術語查卡是否存在 → 缺卡清單。常見模式是「方法論長文存在、原子卡缺席」——術語有整篇長文深入分析、文章引用時連到長文；依賴有宣告、但讀者只要 30 秒定義時被丟進 200 行的文章。這也算缺卡：建原子卡、卡內連長文作深讀路徑。
3. **優先序**：出現篇數 × 支撐角色。高頻支撐術語先建、主線術語即使低頻也建（它是某篇的核心機制、跨文引用價值高）。
4. **批次建卡**：格式與批次紀律走 card-authoring reference。
5. **登記**：新卡進 collection index 分類表；新目錄同步 lint 工具的作用域清單與上層入口——索引是必經註冊點、登記漏了不產生任何錯誤訊號。
6. **回填連結**：每篇文章該術語第一次出現處連卡。回填是整個流程工作量最大的一步（一個術語可能散在十多篇）——先 grep 定位各篇第一次出現、逐處判斷角色（主線不改為連卡、支撐補連結、既有的連長文改連卡）再動手。
7. **主線術語檢查**：各篇的核心機制詞是否行內展開（矩陣第一列）。

## 作用域幻覺：檢查通過可能只是不在作用域

卡片目錄的機器檢查（frontmatter 必填、鄰卡連結、orphan 偵測）通常由 lint 工具的**目錄清單**驅動——frontmatter 檢查跟結構檢查可能是兩個獨立清單。目錄沒登記時、跑檢查會得到零 error——那是「沒被看」、不是「沒問題」。

實測教訓：一批新卡通過了全站卡片檢查、實際上該目錄根本不在檢查作用域內——通過是幻覺。audit 開始前先確認目標目錄在哪些檢查的作用域內；「這個目錄從未在檢查輸出裡出現過」是作用域缺口的訊號、用一個已知違規測試它報不報錯。

## 跨模組大規模重構的執行順序

多個模組的卡片目錄要統一納管時、順序決定成本：

1. **先盤點**：每個目錄的卡數、frontmatter 完整度、結構段（概念位置）存在率、標題命名格式——用 grep 統計、不憑印象。盤點結果把目錄分成「已合規」（直接登記）跟「要先補」（結構或命名不合格）。
2. **先補結構、再登記作用域**：把不合規目錄直接登記進 lint 作用域會讓 CI 大量報錯、迫使用 `--no-verify` 繞過——順序反了會讓工具信譽受損。逐目錄補齊（結構段、命名格式、鄰卡連結）、補完一個登記一個。
3. **最後跑缺卡 audit**：存量目錄合規後、才對各模組的文章跑詞頻掃描補新卡——否則新卡混在舊債裡、審查焦點失散。
4. 每個目錄完成後在重構清單上記錄狀態——跨模組重構是長週期工作、狀態不記錄就會出現「哪些做過了」的考古成本。

## 規模化執行：代理人派工協議

建卡與回填是高度協議化的工作、適合派給較低成本的模型代理執行——判斷已由矩陣承擔、代理只需要「讀 skill → 執行」。實測驗證過的派工要點：

- **prompt 只給工作清單與底線規則、格式判準指向 skill**——skill 寫得夠完整時、代理的判斷品質可靠（主線豁免、標題不連改連正文、錨文字排除這類邊緣案例都能判對）
- **事實紀律要在 prompt 顯式重申**：例子從既有文章 grep 真實數據並標年份、找不到就用通用機制例、禁止編造數字——這是代理執行知識性任務最需要防的失誤面
- **回報必須含完成度自證**（「處理了 N/M 篇」＋逐項統計）——實測的主要失誤模式是代理靜默停止、不回報完成度；驗收方看到「已完成」三個字不可當真
- **驗收以 repo 狀態為準、不以代理報告為準**——用機械檢查驗收（連結計數、重複偵測、標題誤連掃描、lint），報告只用來理解判斷過程
- **代理可能自行再拆批平行**——多代理同時編輯期間、主線不要提前 format / commit（會跟仍在飛行中的編輯交錯）；等全部完成訊號再收尾
- **要求代理回報「skill 規則不清楚或衝突之處」**——每次派工同時是 skill 完整性的測試、回饋直接回流版本
