這一章補的是一個服務一旦進入高併發後最容易被忽略的面向：錯誤處理與測試其實不是兩件事，而是一套可靠性機制。你能不能清楚面對失敗，決定了系統在壓力下是不是還能被理解、被修復、被持續交付。

## 本章目標

學完本章後，你將能夠：

1. 把錯誤處理看成可靠性的一部分
2. 區分可恢復錯誤與不可恢復錯誤
3. 用測試保護失敗路徑與並發路徑
4. 讓 timeout、取消與 race condition 能被提早發現
5. 理解為什麼高併發服務更需要明確的測試邊界

---

## 【觀察】錯誤不是例外，而是服務常態

在高併發服務裡，錯誤不是少見事件，而是日常情況的一部分。網路會失敗、下游會超時、資料會不完整、狀態會競爭，這些都不是異常，而是你必須設計進系統裡的常態。

Go 把錯誤放在回傳值中，就是要讓這些常態能被直接看見。

## 【判讀】測試要先保護脆弱邊界

高併發服務最容易出問題的地方，通常是：

- HTTP handler 與外部輸入邊界
- goroutine 之間的共享狀態
- timeout 與 cancellation
- event / queue 的重複或漏處理

這些邊界都應該有測試，而不是只靠人工驗證。

## 【策略】錯誤路徑也要被測

一個常見問題是測試只驗證成功路徑，卻不驗證失敗路徑。對服務來說，這是不夠的，因為真正會拖垮系統的往往就是失敗時的行為。

你至少應該測：

- 參數不合法時是否回傳穩定錯誤
- 下游失敗時是否有正確的包裝錯誤
- timeout 是否真的會停止工作
- 取消 context 後 goroutine 是否退出

## 【執行】並發測試要看資源是否被正確回收

高併發測試不只是跑很多 goroutine，而是確認：

- goroutine 會不會 leak
- channel 會不會卡住
- 鎖的範圍是否合理
- 資源關閉後流程是否停止

如果測試只能證明「看起來可以跑」，卻不能證明「會正確結束」，那它對服務可靠性的幫助有限。

## 小結

高併發服務需要的不是更多測試數量，而是對錯誤、取消、超時與並發生命週期有更好的保護。Go 的顯式錯誤模型與直接的 testing 風格，正好讓這些可靠性問題更容易被提前看見。
