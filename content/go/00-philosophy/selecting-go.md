這一章不是在比較語言優劣，而是在回答一個更實際的問題：你的工作場景是否值得用 Go 來做。當系統需要長時間運行、明確邊界、穩定併發與簡單部署時，Go 通常是合理選擇；當工作更依賴框架模板、快速表單 CRUD、動態行為或大量 runtime magic 時，先選 Go 反而可能增加成本。

## 本章目標

學完本章後，你將能夠：

1. 判斷哪些工作負載適合 Go
2. 看出哪些系統型態特別適合 Go
3. 區分 Go 的強項與不適合硬上的場景
4. 用工程條件而不是語言情感來做選型
5. 為後續語法與實作章節建立正確的閱讀順序

---

## 【觀察】先看工作負載，再看語言

Go 最常被選中的場景，不是因為它能做所有事，而是因為它很適合一類很常見的服務工作：

| 工作型態           | 為什麼適合 Go                                |
| ------------------ | ------------------------------------------- |
| 高併發 I/O         | goroutine 成本低，適合大量等待型工作         |
| 長連線服務         | 容易管理生命週期、取消與資源清理             |
| 背景 worker        | 可以把工作拆成小單位並持續處理               |
| 事件處理           | channel、select 與明確邊界很適合事件流       |
| API service        | 標準庫直接支撐 HTTP、context、log 與 timeout |

如果工作本質就是一堆等待外部 I/O 的操作，Go 往往比把整個問題放進單線程迴圈更自然。

## 【判讀】架構邊界是否清楚

Go 特別適合邊界清楚的後端服務。當一個系統可以自然拆成輸入、協調、狀態、輸出幾層時，Go 的 struct、interface、package 與明確依賴會讓責任更容易看見。

例如以下這類系統通常是 Go 的好候選：

- WebSocket 即時服務
- notification service
- queue consumer
- log / event pipeline
- 需要清楚 ports/adapters 的 backend service

相反地，如果你的產品高度依賴某個框架提供的大量現成功能，或核心價值在於快速拼接大量業務頁面與模板，那 Go 不一定是最省力的起點。

## 【策略】runtime 與部署條件也是選型的一部分

Go 的優勢不只是語法，還包括 runtime 與部署形態：

- 單一 binary，部署流程簡單
- 啟動速度快，適合 container 與短週期交付
- 標準庫完整，很多服務不需要先找一堆框架
- 可讀性高，長期維護成本較容易控制

如果你的團隊很在意：

- 記憶體用量
- 啟動時間
- 觀測與除錯
- 服務在高流量下的穩定性

那 Go 的工程價值就會很明顯。

## 【執行】哪些情況不建議先選 Go

不是每個後端問題都應該先用 Go 解。以下情境通常不適合把 Go 當第一選擇：

| 情境                   | 原因                                               |
| ---------------------- | -------------------------------------------------- |
| 極度偏 CRUD 模板系統   | 框架生態可能比語言特性更重要                       |
| 大量動態行為與 runtime 配置 | Go 會要求更多顯式設計                              |
| 團隊主要目標是快速試錯 | Go 的工程紀律可能比腳本型語言更有前置成本         |
| 主要瓶頸在前端整合流程 | 語言效能不是核心解題點                             |

這不是說 Go 做不到，而是說它不一定是最符合成本效益的方案。

## 小結

Go 適合的是工作負載明確、邊界清楚、需要高併發與穩定維護的服務型系統。先把選型條件講清楚，後面的語法與實作才不會變成「學會了工具卻不知道何時使用」。
