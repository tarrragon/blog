本地 LLM 最大的挫折來源不是「速度慢」「裝得久」，而是「期待錯了」。把本地當作 Claude / GPT 替代品，會在第一週就因為它跨檔案重構失敗、規劃 multi-step 任務崩潰、深度 debug 給出錯誤建議而失望。但若把它當作「免費、永遠在線的初階 pair programmer」，同一套行為會變成可接受、甚至好用。

本章的責任是把期待校準到現實。讀完後你應該能清楚知道：哪些任務交本地、哪些交雲端、本地 LLM 一週後該留還是該卸載、什麼時候硬體升級才有意義。

## 本章目標

讀完本章後，你應該能：

1. 用「能 / 不能」二分法清楚知道每類任務交給誰。
2. 建立「本地 vs 雲端」的切換反射，不再每次糾結。
3. 用一週實測決定本地 LLM 是否留下來。
4. 識別本地 LLM 不是萬靈丹但也不該卸載的場景。

## 本地能做什麼：明確強項

本地 LLM 在這些任務上的表現「足夠好、足夠快、值得每天用」：

| 任務                                | 為什麼適合本地                   |
| ----------------------------------- | -------------------------------- |
| 補 type annotation                  | 模式單純、context 短、本地速度快 |
| 寫 docstring                        | 模式單純、有現成函式可看         |
| 寫 unit test 第一版                 | 任務有結構、可以邊讀邊修         |
| 解釋程式碼片段                      | 短 context、不需要跨檔案推理     |
| 改名變數 / 函式（refactor rename）  | 任務範圍明確、不需要創造力       |
| 把 callback 改成 async/await        | 常見 pattern、模型訓練資料多     |
| 把 for loop 改成 list comprehension | 同上                             |
| 寫 SQL（簡單 query）                | 有明確語法、可以邊跑邊改         |
| Git commit message                  | 任務簡短、本地隱私夠             |
| 寫 README / changelog 草稿          | 草稿後人類會修，品質要求中等     |
| 解釋錯誤訊息                        | 多半是已知 pattern               |
| 把 JSON / YAML 轉換格式             | 任務機械化                       |

這份清單覆蓋了一般工程師每天 60 ~ 80% 的 LLM 使用情境。把這些餵給本地，雲端 API 費用會大幅下降。

## 本地不能做什麼：明確弱項

本地 LLM 在這些任務上的表現「明顯弱於雲端旗艦、會浪費你的時間」：

| 任務                                   | 為什麼本地不行                                |
| -------------------------------------- | --------------------------------------------- |
| 跨多個檔案的重構                       | context window 限制 + 推理深度不足            |
| 設計新模組的架構                       | 需要綜合判斷、雲端旗艦深度領先                |
| 規劃 multi-step 任務（拆 todo）        | 規劃能力是雲端旗艦的明顯強項                  |
| 深度 debug（非常見錯誤）               | 需要推理能力與大量訓練資料                    |
| 評估技術選型（A vs B）                 | 需要廣泛知識與權衡能力                        |
| 寫長篇技術文件                         | 篇幅大、邏輯連貫要求高                        |
| 從模糊需求拆出 acceptance criteria     | 需要產品意識，模型訓練資料中較少              |
| 數學推理（複雜演算法）                 | 雲端旗艦的 reasoning effort 模式領先明顯      |
| 解少見語言（COBOL、Erlang）            | 訓練資料少，hallucination 機率高              |
| 處理長 context（10K+ tokens）          | 本地 TTFT 太痛，雲端旗艦的 context 處理更穩定 |
| Agent 模式（複雜 multi-step tool use） | 本地 tool use 支援陽春，雲端 agent 框架成熟   |

這份清單覆蓋了「LLM 真正取代人類思考的部分」。本地 LLM 在這部分還做不到。

## 模糊地帶：要試了才知道

下列任務本地有時行有時不行，建議「先試本地、不行再切雲端」：

| 任務                               | 嘗試本地 → 切雲端的觸發訊號   |
| ---------------------------------- | ----------------------------- |
| 解釋一個 bug 的根本原因            | 本地給出來的解釋說不通 / 含糊 |
| 改寫一段較複雜的 function          | 改完後測試失敗、邏輯偏了      |
| 寫一段中等長度（< 50 行）的新 code | 跑不過或明顯結構不對          |
| 翻譯 code 註解到另一種語言         | 翻得不自然或失準              |
| 寫單元測試（中等複雜度的函式）     | 測試覆蓋不到關鍵分支          |
| 回答一個技術概念性問題             | 答案模糊 / 跟你已知不符       |

這個分類的核心啟示是：**不要對本地 LLM 失敗感到挫折**。模糊地帶的失敗是預期內的；切到雲端是正常工作流，不是本地「失敗」。

## 切換的具體流程

Continue.dev 的 chat panel 下方有 model selector，可以直接切。建議的反射動作：

1. **預設用本地**：開啟 Continue panel 時，先選本地 model。
2. **碰到弱項任務直接切**：上面表格的「不能」類任務，第一次提問就選雲端 model。
3. **模糊地帶試一次本地**：如果本地的回答不堪用，立刻切雲端重提同樣 prompt。
4. **記下你的本地 hit rate**：用一週，記錄哪些任務本地成功、哪些失敗。第二週開始就有自己的判斷依據。

不要陷入「我裝了本地、就要硬用」的沉沒成本陷阱。本地是工具不是身份，能用就用、不能用就切。

## 用一週實測：去留決策

裝完本地 LLM 後，建議用一週實測決定是否留下來。實測要做的事：

1. **每次用 LLM 都先試本地**。不要繞過本地直接用雲端。
2. **記錄 hit rate**。簡單試算表，欄位：任務描述、本地成功 / 失敗、切雲端後成功 / 失敗。
3. **記錄體感速度**。本地是「快得舒服」「等得心煩」哪一邊。
4. **記錄記憶體與發熱**。Mac 是否變慢、風扇是否狂轉影響其他工作。

一週後做決策：

| 觀察結果                                 | 建議                                        |
| ---------------------------------------- | ------------------------------------------- |
| Hit rate > 60%、體感速度可接受、Mac 沒崩 | **留下**，本地是日常主力                    |
| Hit rate 40 ~ 60%、體感速度可接受        | **留下**，但混用雲端要更積極                |
| Hit rate < 40%                           | **可能該卸載**或換更大模型                  |
| 體感速度太慢（< 10 tok/s）               | 換較小模型或考慮升級硬體                    |
| Mac 持續變慢、風扇狂轉                   | 記憶體不足，換較小模型或承認 Mac 不適合本地 |
| 雲端 API 費用沒降                        | 你沒真的切到本地、回去檢查切換習慣          |

這個實測比看 benchmark 重要得多，因為你的工作流跟 benchmark 設定的任務分佈不一樣。

## 「本地 LLM 是免費的初階 pair programmer」

把本地 LLM 定位成「免費的初階 pair programmer」，期待會自然對齊：

1. **初階 pair programmer 是有用的**。能寫測試、能解釋程式碼、能補 type、能改 callback。這些事一個 junior 同事每天做得很好。
2. **初階 pair programmer 有限制**。設計新架構、跨檔案重構、評估技術選型，junior 同事做不到，要交給 senior（雲端旗艦）。
3. **初階 pair programmer 隨時在線、不用付薪水**。這是本地 LLM 比 junior 同事還好的地方。
4. **初階 pair programmer 不能取代 senior**。但能讓 senior 把時間花在真正困難的部分。

陷阱是把本地當「便宜的 senior」。它真的不是；它是免費的 junior。明確這個定位後，你會自然把日常雜事丟給本地、把難題留給雲端。

## 跟雲端旗艦的協作姿勢

「混用」不是「隨機切換」，而是有結構的協作姿勢：

| 場景                       | 流程                                                         |
| -------------------------- | ------------------------------------------------------------ |
| 我有個新 feature 要開發    | 雲端旗艦規劃 → 本地寫 boilerplate → 雲端旗艦審 critical 部分 |
| 我要 debug 一個 bug        | 本地解釋錯誤訊息 → 自己看 code → 雲端旗艦審 root cause       |
| 我要重構一個 module        | 雲端旗艦設計新結構 → 本地實際改 code → 雲端旗艦審差異        |
| 我要寫一份技術文件         | 雲端旗艦寫大綱 → 本地寫各節草稿 → 自己潤稿 → 雲端旗艦審稿    |
| 我要寫測試                 | 本地寫 → 自己跑 → 缺漏處交雲端旗艦補                         |
| 我要 commit                | 本地寫 commit message、自己審                                |
| 我要解釋一段 code 給同事看 | 本地寫解釋、自己審                                           |

這個結構讓「雲端旗艦的高品質」用在最值錢的地方（規劃 + 審稿），「本地的免費 + 速度」用在批量產出。雲端 API 費用會大幅下降，思考品質仍然維持高水準。

## 什麼時候該升級硬體

裝完本地、用一週後，可能會想「升級 Mac 是否值得」。判斷依據：

1. **記憶體不夠**：跑 14B 模型體感卡 → 升 24GB；跑 31B 卡 → 升 32GB；跑 70B 卡 → 升 64GB。
2. **生字速度不夠**：用最強硬體還 < 10 tok/s 表示模型太大，要換較小，不是升級硬體。
3. **Hit rate 太低**：問題不在硬體，在於本地模型能力不夠，升級無解。
4. **想跑 long context**：升級到 48GB+ 才能處理 16K+ context。

陷阱是把「想升級」混在「正當理由」裡。先用一個月再決定；多數情況下省下的 API 費用攤平不了升級成本。

## 識別「該卸載」的訊號

下列訊號表示本地 LLM 在你身上沒用，可以乾脆卸載：

1. 一週後雲端 API 費用沒降，因為你還是習慣用雲端。
2. 本地回答太慢，你已經放棄用、但 Ollama 還在背景吃記憶體。
3. 你的 Mac 規格本來就太小，跑本地讓其他工作變慢。
4. 你的工作主要是規劃、設計、複雜推理，本地的弱項剛好是你的主場。

卸載不是失敗。本地 LLM 不適合所有人；正確判斷它對你的工作流沒幫助，是合理結論。

```bash
# 完整卸載 Ollama
brew services stop ollama
brew uninstall ollama
rm -rf ~/.ollama

# 卸載 Continue.dev 擴充套件
# 在 VS Code Extensions panel 找到 Continue 點 Uninstall
rm -rf ~/.continue
```

卸載後可以雲端 API 全用 Claude Code、Cursor 或其他雲端 IDE plugin，不會有任何遺憾。

## 小結

本地 LLM 的正確心態是「免費的初階 pair programmer」。它在簡單、重複、隱私敏感的任務上強，在複雜規劃、跨檔案、深度 debug 上弱。期待對齊現實後，「混用本地 + 雲端」是現階段最穩的工作姿勢；用一週實測決定去留，不要陷入沉沒成本陷阱。
