組裝層是應用程式把 [port](/ddd/knowledge-cards/port/) 插上 [adapter](/ddd/knowledge-cards/adapter/) 的地方——port 是 domain 對外宣告的介面、adapter 是介面的具體實作。依賴注入的組裝集中在 [composition root](/ddd/knowledge-cards/composition-root/)（組裝根、唯一的組裝起點），路由表以單點註冊靠攏它，UI 事件的接線散佈在各畫面：位置不同、同屬組裝層的責任。本模組（[DDD 領域驅動設計指南](/ddd/)）的源頭句「讓違反規則的路徑走不通」守的是領域內部的規則；本章守它的對偶面——讓正確的路徑確實走得通。domain 模型全對、系統不可用，是分層架構特有的失效形態：每一層各自正確、層與層之間沒接上。層內的設計判準（[資料袋與領域模型](/ddd/data-bag-vs-domain-model/)、[不變式的強制層次](/ddd/invariant-enforcement-layers/)）決定六角形架構（hexagonal，即 ports & adapters）內部的品質。[建構路徑設計](/ddd/construction-path-design/) 收在物件怎麼被建出來；本章把同一個問題抬到應用程式層——整個系統怎麼被組起來、六角形的邊界有沒有被接進真實世界。

## 分層的承諾與它的影子

分層架構（以及 ports & adapters）的核心承諾是 domain 不依賴框架、可以脫離 infrastructure 單獨測試。兌現承諾的機制是測試 seam：介面加上注入點，測試時把真實依賴換成 mock。seam 工作得越好，domain 的測試越乾淨——這是 DDD 教材反覆強調的部分。

承諾有一道影子。mock 換掉的正是組裝，於是「組裝有沒有完成」在以 seam 為基礎的測試套件裡沒有任何證言——證言、指一個測試通過時能證明的事。兩類測試各自的證言範圍：

| 測試     | 執行環境                              | 證明什麼            | 對組裝的證言 |
| -------- | ------------------------------------- | ------------------- | ------------ |
| 行為測試 | mock／override 注入                   | 功能邏輯正確        | 零           |
| 接線測試 | 組裝路徑零 override、真實路由表與依賴 | port 插上了 adapter | 唯一來源     |

接線測試（wiring test）在既有測試光譜上的位置：它是端對端測試的窄化——E2E 在目標平台同時驗行為與接線，代價是裝置需求與執行不穩定；接線測試只驗「插上了沒」，換取在 host 環境快速且確定地執行。它也是 DI 容器自驗證（啟動時逐一解析全部註冊項的那類機制）的推廣：解析之外，把路由與 UI callback 的接線納進同一種證言。零 override 指的是組裝路徑——domain 到入口之間不替換任何一段；最外圈的 infrastructure（遠端服務這類）可以在邊界替換，不影響組裝的證言。

一個書籍管理 App 把這道影子完整走了一遍：單元測試全數通過，實機測試找出五個問題，其中三個是組裝斷裂——DI 容器的 provider 停在佔位 throw、路由表把入口指向佔位頁、按鈕的 callback 是空實作（[測試全綠、功能失聯](/work-log/flutter_composition_root_wiring_gap/)）。功能單元全部存在、行為測試全綠；斷的每一處都在組裝層，而那正是測試套件沉默的地方。

## 佔位：合法的中間態、比文件層更靜默的失效

佔位（指向「開發中」頁的路由、空 callback、throw「requires override」的 provider）是開發過程的合法中間態——先把介面立起來、實作後補，這個節奏本身沒有問題。問題在佔位的失效形態。[不變式的強制層次](/ddd/invariant-enforcement-layers/) 記過文件層約束的失效是靜默的；佔位的失效比文件層再隱蔽一級——它讓測試綠燈。型別層看佔位是合法構件，行為測試的 override 讓它永遠不會被觸發，只有零 override 的解析會撞上它。文件層約束至少留下「規則寫在那裡、沒人遵守」的對照證據；佔位連對照證據都沒有，驗收讀到的只有通過。

佔位因此需要一個獨立的攔截點。這個案例的處置是把攔截放進發版前置檢查：出現過的三種形態（佔位路由、佔位 provider、空 callback）都是靜態可掃描的，掃到即警告、由人判斷是刻意中間態還是漏網。回傳假值的佔位（hardcoded 假資料、回空集合的 stub）在掃描與接線測試眼中都是正常構件——provider 解析得了、導航也會發生——要靠行為斷言或冒煙清單攔。

## 可達性作為組裝層的不變式

「use case 描述的每個入口在 production 環境可達」是一條不變式。套用文件／型別／執行的三層判準問它該落在哪一層強制，答案是三層都有位置、單獨任何一層都不夠：

**文件層**：use case 的成功保證補可達性條款（路由指向真實頁面、callback 已接線、provider 在無 override 環境可解析）。這層的作用是把組裝責任寫進驗收範圍——它定義「什麼叫完成」，但文件層失效靜默，單獨存在時只是願望。

**型別層**：組裝完成性用型別表達的空間有限。DI 的解析多數發生在 runtime，編譯器看不見「這個 provider 沒有人提供實作」——compile-time DI 生態是例外，runtime 解析的容器（Riverpod 這類）做不到。路由表以字串鍵對頁面的生態同樣把斷鏈推遲到執行期。型別層在這條不變式上的角色是輔助而非主力。

**執行層**：接線測試——組裝路徑零 override 的環境逐一解析每個注入項（provider）、真實路由表逐條導航、每顆按鈕實際點擊斷言導航發生。違反當場紅燈，失效點集中。刻意不可達的入口（feature flag 關閉、分階段發布）不算違反：以旗標開啟的組態跑接線測試、或明列豁免清單——豁免要有記錄。書籍 App 的修復批次先補了這批測試，案例對目標行為斷言、修復前確定性紅燈，修復以測試變綠為驗收點。

**發版層**（對應 [不變式的強制層次](/ddd/invariant-enforcement-layers/) 補充的資料庫約束與 CI 檢查層）：佔位掃描加實機冒煙清單。前者攔靜態可查的佔位；後者攔 host 測試構不到的平台語意差異——這類缺口在組裝層之外，分類與對策見「邊界」一節。

## use case 的完成定義

組裝斷裂的上游在設計文件。修復批次反向追溯五個問題對應的提案與 use case，發現一個一致的形狀：提案思考「能力」（系統要有哪些功能）、use case 思考「行為」（使用者做什麼、系統回應什麼），兩者之間的「裝配」——把能力組起來、給行為一個入口的責任——沒有歸屬。三個組裝斷裂全部落在這條縫裡。

把縫補起來的做法是讓 use case 撰寫承擔對提案的壓力測試。檢核問句從五個問題歸納而來、是起點集而非全集，新的斷裂形態出現時往上加：

| 檢核問句                                                                                         | 填不出來代表                 |
| ------------------------------------------------------------------------------------------------ | ---------------------------- |
| 名詞可定位：文中每個畫面／按鈕／服務，能填出「位於哪個畫面、經哪條路由、由誰供給」嗎             | 提案只設計了能力、沒設計裝配 |
| 路徑連通：步驟一的入口能從應用程式啟動畫面一路追到嗎                                             | 入口接線無人認領             |
| 狀態完備：涉及的每個畫面，狀態機的每個狀態（空／正常／錯誤、權限與登入態）下入口可見性都有答案嗎 | 畫面狀態機未被枚舉           |
| 環境差異：涉及平台 API 的步驟，「測試環境與目標平台行為一致嗎」有答案嗎                          | 平台敏感點未被標記           |

任何一問填不出來，代表對應的設計缺口還停在提案層，退回補齊再進實作；第四問答案為否或不確定的步驟，標記「需實機驗證」、餵給發版前的冒煙清單。

第三問附帶一個操作型式：前置條件反向展開。use case 的前置條件（「書庫中存在至少一本書」）每一條都隱含一個「不滿足時會怎樣」的狀態分支；前置條件被讀成「範圍外不歸我管」的免責聲明時，不滿足的分支就無人設計——案例裡的匯出按鈕只存在於正常狀態分支，空書庫的使用者看不到它、回報「功能不見了」。前置條件是狀態枚舉的線索，與 [狀態轉換與稽核軌跡](/ddd/state-transition-and-audit-trail/) 談的領域狀態機互為投影：領域的前置條件投影到畫面層，就是「這個狀態下入口在哪」的答案。

## 邊界：組裝可達之外的失效

缺口分類先於對策。五個問題裡的兩個平台差異（框架初始化順序、資料庫 PRAGMA 在 Android 的執行語意）在組裝層之外——設計文件構不到、host 測試也構不到，對策是在設計期標記「需實機驗證」、餵給發版前的冒煙清單。組裝斷裂修規格與測試、平台差異修驗證流程；混為一談會把「補測試」當成萬靈丹，而平台語意差異是 host 測試補不到、要目標平台實際執行才暴露的那一類。

本章的作用域同樣要誠實標明：可達性守的是「入口接上了」，接上之後的行為正確性屬行為測試、資料在層間流動的正確性屬整合測試的其他主題。組裝層的證言與六角形內部的證言互補、彼此不可替代——這正是本章與模組其他章的關係：層內的模型設計得再好，組裝層沒有證言，使用者拿到的仍然可能是一片接不通的綠燈。
