UX 設計教材的核心目標是教讀者用系統性方法捕捉 UX 設計缺口，而不是靠實機測試才發現「使用者被困住」。加 back 按鈕是 5 分鐘的事，但問題不在按鈕 — 在企劃階段沒有工具強制你為每個畫面的每個狀態想退出路徑。本教材把 UX 設計從「靠經驗想到」變成「靠方法查到」。

## 教學出發點

這個系列從一個具體事件出發：一個 Flutter app 的 Terminal 畫面有五個狀態（idle、connecting、connected、error、disconnected），五個狀態零個退出路徑。使用者一旦進入 Terminal 畫面就出不去，唯一的出路是殺掉 app。

企劃文件的操作盤點確實列了「連線失敗顯示無法連線」這個失敗情境，但「前端引導」只寫了一句話 — 它覆蓋了 `error` 狀態的**顯示**，但沒有回答**操作**（重連？返回？）和**退出**（怎麼離開這個畫面？）。

這揭示的是一個方法缺口：**從 BDD 操作盤點到 UI 實作之間，缺少一個系統性的展開步驟 — 把操作情境轉換成畫面狀態、可用操作和退出路徑的矩陣。**

起源事件之後案例庫持續擴充，目前涵蓋 mobile app 與 Chrome extension 兩類 surface 共 20 個實戰案例 — 入口在 [UX 設計案例庫](/ux-design/cases/)。

## 教學範圍

本系列聚焦「工程師能自己執行的 UX 設計方法」，不討論視覺設計、品牌、色彩理論或使用者研究方法論。

| 放在本系列                                                            | 放在其他系列                                                                               |
| --------------------------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| 畫面狀態機設計（狀態 × 操作 × 進入 × 退出矩陣）                       | 視覺設計（色彩、字型、間距）                                                               |
| 導航模式（mobile 的 push/pop/go、tab、drawer）                        | Design system 視覺實作（token / 元件庫，放語言教材或 frontend 系列；按鈕狀態語意在模組六） |
| Gate fallback 設計（biometric、network、auth 的失敗替代路徑）         | 使用者研究方法（訪談、問卷、A/B test）                                                     |
| 輸入機制設計（keyboard type、submit model、IME policy、special keys） | 動畫與 motion design                                                                       |
| 錯誤狀態與回復（error → retry / back / alternative）                  | Accessibility 標準（放獨立系列或本系列後續模組）                                           |
| 使用者行為驗證（畫面狀態 coverage test、導航路徑 test）               | 效能最佳化本體（真實延遲的量測與改善）                                                     |
| 互動回饋與感知效能（點擊確認、loading、skeleton、optimistic update）  |                                                                                            |

## 與 Testing 的關係

[開發測試](/testing/) 教材的模組四（自動化 UI 驗證）聚焦「怎麼用 test 驗證 UI 行為」。本系列聚焦「怎麼設計 UI 行為讓它可被驗證」。兩者的交叉點是畫面狀態矩陣 — 設計端產出狀態 × 操作 × 退出的表格，測試端依這張表寫 widget test 覆蓋每個狀態。

## 教學模組

### 模組一：畫面狀態機設計

回答「這個畫面有幾個狀態、每個狀態能做什麼、怎麼離開」。核心工具是**畫面狀態矩陣**：

| 畫面.狀態             | 顯示            | 可用操作     | 進入條件       | 退出路徑                        |
| --------------------- | --------------- | ------------ | -------------- | ------------------------------- |
| Terminal.idle         | 空白            | —            | 從首頁導航進入 | back → 首頁                     |
| Terminal.connecting   | 進度指示        | —            | 自動觸發連線   | back → 首頁（取消連線）         |
| Terminal.connected    | 終端機 + 工具列 | 打字、特殊鍵 | WS 連線成功    | disconnect → idle；back → 首頁  |
| Terminal.error        | 錯誤訊息        | 重新連線     | 連線失敗       | back → 首頁；retry → connecting |
| Terminal.disconnected | 「連線中斷」    | 重新連線     | WS 斷線        | back → 首頁；retry → connecting |

**退出路徑欄位為空 = UX 死胡同。** 這張表格在 10 分鐘內暴露所有導航缺口，省掉實機測試才發現的成本。

**從操作盤點到狀態矩陣的展開步驟**：

1. 從 BDD 操作盤點列出所有畫面
2. 每個畫面列出所有狀態（enum 值或邏輯分支）
3. 填畫面狀態矩陣的四個欄位
4. 檢查退出路徑欄位 — 空格 = 死胡同
5. 每個 gate 標注 fallback

> 章節入口：[模組一：畫面狀態機設計](/ux-design/01-screen-state-machine/)
>
> 案例入口：[每個畫面都需要出口](/work-log/ux_screen_state_machine_design/) — Terminal 畫面五個狀態零個退出路徑

### 模組二：Gate 與 Fallback 設計

回答「使用者過不了關卡時怎麼辦」。Gate = 使用者必須通過的關卡（biometric、network、auth、permission）。

| Gate 類型  | 設計問題                                   | 反模式                               |
| ---------- | ------------------------------------------ | ------------------------------------ |
| Biometric  | Face ID 不可用（光線、口罩、設備）→ 替代？ | `biometricOnly: true`（無 fallback） |
| Network    | VPN 離線、DNS 失敗 → 替代？                | 直接顯示錯誤碼（使用者看不懂）       |
| Auth       | Token 過期、密碼錯誤 → 替代？              | 靜默失敗（使用者不知道發生什麼）     |
| Permission | 相機/位置權限被拒 → 替代？                 | 只說「需要權限」不說為什麼           |

每個 gate 的設計至少要回答三個問題：成功時做什麼？失敗時做什麼？使用者不知道發生什麼事時做什麼？

> 章節入口：[模組二：Gate 與 Fallback 設計](/ux-design/02-gate-fallback/) — biometric fallback、網路斷線 UX、permission 請求時機等章節已上線

### 模組三：輸入機制設計

回答「使用者怎麼輸入資料」。特別針對 mobile 環境，手機鍵盤和桌面鍵盤的差異比想像的大。

| 設計維度      | 桌面預設               | Mobile 需明確設計                                              |
| ------------- | ---------------------- | -------------------------------------------------------------- |
| Keyboard type | 實體鍵盤，所有鍵都有   | 虛擬鍵盤，需選 type（text / number / email / visiblePassword） |
| 自動校正      | 通常關閉或可忽略       | 預設開啟，可能「幫忙」修改 CLI 命令                            |
| Special keys  | Esc / Tab / Ctrl 都有  | 沒有，需要自訂工具列                                           |
| Submit        | Enter 換行或送出       | 需明確選擇 `textInputAction`（send / done / next / newline）   |
| IME           | 輸入法不影響程式碼輸入 | 中文輸入法的建議列可能遮擋畫面                                 |

這些決策影響 UI layout 和 protocol 設計，應該在企劃階段做，不是實作時臨時決定。

> 章節入口：[模組三：輸入機制設計](/ux-design/03-input-mechanism/) — terminal 輸入設計、表單與搜尋 UX 模式等章節已上線

### 模組四：錯誤狀態與回復

回答「出錯時使用者能做什麼」。錯誤不只是紅色文字 — 是一個需要設計退出路徑的狀態。

> 章節入口：[模組四：錯誤狀態與回復](/ux-design/04-error-recovery/) — 錯誤訊息撰寫、retry 機制、degraded mode 等章節已上線

### 模組五：導航模式

回答「畫面之間怎麼跳」。Mobile 導航模式的選擇（push/pop stack、GoRouter 命名路由、tab bar、drawer）和每種模式的 UX 差異。

> 章節入口：[模組五：導航模式](/ux-design/05-navigation-patterns/) — GoRouter 導航、iOS 與 Material 導航差異、deep link 等章節已上線

### 模組六：互動回饋設計

回答「使用者按了按鈕之後發生什麼事」。核心原則：每個使用者動作都需要即時、誠實的系統回應 — 沒有回饋的按鈕等同壞掉的按鈕。

- 互動回饋三層模型（點擊確認 / 等待指示 / 結果通知 + 畫面級狀態轉換 + 結果通知的鏈路前提）
- 按鈕狀態設計（default / hover / active / focus / disabled / loading）
- 元件語意與版面（切換標籤的現態 / 動作歧義、可點性辨識、選中態對比、溢出 affordance、回饋的版面保障）
- 時間感知與回應策略（100ms / 400ms / 1s / 10s 門檻、spinner vs skeleton 選擇、完成宣告的窮盡證據）
- 通知模式選擇（SnackBar / Dialog / Banner / Bottom Sheet 的干擾程度判準）

> 章節入口：[模組六：互動回饋設計](/ux-design/06-interaction-feedback/)

## 學習路線

| 路線            | 適合讀者                                                                                                     | 建議順序                                            | 讀完能做什麼                                 |
| --------------- | ------------------------------------------------------------------------------------------------------------ | --------------------------------------------------- | -------------------------------------------- |
| UX 防呆入門     | 想避免「使用者被困住」的基本問題                                                                             | 模組一 → 模組二 → 模組四                            | 能用狀態矩陣捕捉所有 UX 死胡同               |
| Mobile 輸入設計 | 做 mobile app 但鍵盤行為總是出問題                                                                           | 模組三 → 模組一（狀態中的輸入）                     | 能設計完整的 mobile 輸入方案                 |
| 全棧 UX 設計    | 想建立系統性的 UX 設計流程                                                                                   | 模組一 → 模組二 → 模組三 → 模組四 → 模組五 → 模組六 | 能從操作盤點展開完整的 UX 設計文件           |
| 互動回饋設計    | 按鈕按下去沒反應、重複提交、標籤被讀反、選中看不到字、文字被壓成省略號、通知形式選錯、或要為新功能定回饋規格 | 模組六                                              | 能設計完整的按鈕回饋、等待策略和通知模式選擇 |

## 教學寫作方向

本系列的寫作原則：

1. **方法先於工具** — 先教狀態矩陣怎麼填，再討論用什麼 framework 實作。Figma、Sketch、紙筆都能畫狀態矩陣
2. **工程師視角** — 不假設讀者有設計背景。用程式碼的 enum、state machine、if-else 類比解釋 UX 概念
3. **反模式驅動** — 每個設計原則都從一個真實反模式出發（「五個狀態零個退出路徑」），先讓讀者感受問題，再教方法
4. **可測試的設計** — 設計產出物（狀態矩陣）要能直接轉成 test case。設計和測試不是兩件事
