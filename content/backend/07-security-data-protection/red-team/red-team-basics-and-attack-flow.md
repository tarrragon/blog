本章先建立紅隊共同語言。紅隊在本教材中的意思是「攻擊者視角的風險檢查方法」：用真實攻擊路徑反向驗證服務邊界、流程與操作設計。目標不是教 exploit，而是讓團隊在設計階段先看見可被利用的路徑。

## 【判讀】為什麼服務供應方要先懂攻擊流程

服務供應方需要先理解攻擊流程，才能把防護做在正確位置。若只從功能視角設計，常見結果是入口有驗證、流程仍可濫用；資料層有授權、匯出路徑仍可外流；平台有擴容、設定錯誤仍可打開隱藏入口。

攻擊流程的價值是提供「先後順序」：攻擊者先看見什麼、先試什麼、成功後往哪裡移動。這個順序能直接轉成防護優先級。

## 【共同詞彙】紅隊、藍隊、紫隊在教材中的角色

- 紅隊：從攻擊者視角找可利用路徑，驗證系統弱點是否可形成實際風險。
- 藍隊：從防守與維運視角縮短偵測、回應、收斂時間。
- 紫隊：把紅隊發現與藍隊防護放在同一輪驗證，快速修正控制面。

本系列聚焦紅隊視角的設計判讀，並讓藍隊與事故流程可直接接上。

## 【常見流程】攻擊者在服務系統中的六段路徑

以下流程可作為教學與設計檢查的基線：

1. 偵察（Recon）：盤點公開入口、路由、錯誤訊息、版本痕跡與可枚舉資源。
2. 初始進入（Initial Access）：利用弱驗證、錯誤設定、流程缺口或暴露端點取得第一個落點。
3. 權限與持續控制（Privilege / Persistence）：擴大可操作範圍，維持可重複進入能力。
4. 橫向移動（Lateral Movement）：從單一服務移向資料層、內部服務或管理平面。
5. 目標行動（Collection / Exfiltration / Impact）：蒐集資料、外送資料、破壞可用性或操控業務結果。
6. 擾動掩護（Defense Evasion）：降低被偵測機率，延長停留時間。

這六段不是固定腳本，而是常見的決策路徑。每一段都能對應到本模組後續章節。

## 【對應地圖】流程與防護控制面的對齊方式

- 偵察 -> 攻擊面盤點、錯誤回應最小化、入口分級
- 初始進入 -> 身份驗證、授權、輸入驗證、設定治理
- 權限與持續控制 -> 最小權限、稽核、憑證與密鑰輪替
- 橫向移動 -> 信任邊界重驗證、服務間存取策略、網路分段
- 資料外送與破壞 -> 資料分級、遮罩、匯出控制、速率與容量保護
- 擾動掩護 -> 可觀測性關聯、告警品質、事故指揮與復盤

## 【落地方式】本系列如何使用這套流程

後續 `7.R1` 到 `7.R5` 會用同一個判讀順序：

1. 先找入口與邊界。
2. 再找流程濫用與資料路徑。
3. 接著評估容量與設定風險。
4. 最後回到控制面與事故應對，確認可檢測、可回復、可稽核。

這種寫法可以讓設計、驗證與事故流程使用同一張風險地圖。
