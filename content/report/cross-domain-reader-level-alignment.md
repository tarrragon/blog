## 核心原則

跨領域分析的讀者層級要先定位、再決定術語密度。把商業分析寫給工程背景讀者時、不能繼承原分析文章的 VC / founder / industry insider 讀者假設；要把原文的高密度術語與壓縮因果鏈降到目標讀者可複製判讀的層級。

| 維度     | 高階同領域讀者                            | 跨領域教學讀者               |
| -------- | ----------------------------------------- | ---------------------------- |
| 術語     | 可連續使用 CAC / LTV / ARR / gross margin | 首次出現要就地解釋或連到卡片 |
| 因果鏈   | 可跳過中間步驟                            | 每段只推一到兩個因果步       |
| 數字     | 可用百分比與估值倍數暗示                  | 要補公式、分母、比較基準     |
| 文章目標 | 展示判斷深度                              | 教讀者如何重做判斷           |

判別問題：「讀者若沒有商業分析背景，能不能用這段推導去分析下一個事件？」不能，就要降一級。

## 情境

`content/business/reading-frameworks/writing-down-a-level.md` 是從 business case-analyses 的 Round 5 修文抽出來的。當時讀者回饋指出，文章雖然比第一版更像教學，但仍太像寫給 VC / founder 的 deep blog：三句內連續塞進多個商業術語，且因果鏈跨太多步。

例如 Claude for Legal 的第一層擠壓原本把 enterprise license、API margin、長約、垂直流程、法律風險等概念壓在少數句子裡。工程背景讀者可能知道 API、SaaS、workflow，但未必直覺知道「為什麼毛利結構會推動供應商賣 enterprise contract」。若不拆開，讀者只能記住結論，無法複製推導。

後續 `b9c5f06` 新增 reading framework，`8c67ab6` 再把 FDE 與 Bufstream 文章的 5 段高密度內容降一級，形成穩定規則：跨領域文章要先判斷原文寫給誰，再決定本站要降到哪裡。

## 理想做法

### 第一步：先辨識原文的 reader contract

外部商業文章常見 reader contract：

| 原文類型        | 預設讀者                       | 寫作特徵                       |
| --------------- | ------------------------------ | ------------------------------ |
| 大眾財經新聞    | 一般投資讀者                   | 事件摘要、股價、短期影響       |
| VC / founder 文 | 創辦人、投資人、策略工作者     | 術語密度高、暗示多、判斷快     |
| 產業內部文      | 已在該市場工作的人             | 預設供應鏈、客戶型態、競爭格局 |
| MBA 教材        | 有基礎商業概念但不熟特定產業者 | 概念明確、例子完整、因果鏈可追 |

本站 business case-analyses 的目標更接近「工程背景讀者讀 MBA 教材」：保留商業分析深度，但不預設讀者熟悉投資圈 shorthand。

### 第二步：用術語密度決定是否降一級

一段內若連續出現三個以上跨領域術語，就要拆段或補橋接句。

```text
API margin → enterprise ARR → vertical workflow lock-in
```

這串對 VC 讀者可能順，但對工程背景讀者需要拆成：

1. API 型產品的收入跟成本如何變動。
2. 為什麼長約能降低收入波動。
3. 為什麼垂直流程能讓客戶更難替換。

### 第三步：用因果鏈步長決定段落切分

每段只推一到兩步。若一句話需要讀者同時理解「成本結構、銷售模式、採購風險、workflow switching cost」，拆成多段。

| 原句壓縮方式     | 降級後寫法                     |
| ---------------- | ------------------------------ |
| A 導致 B、進而 C | 第一段 A→B，第二段 B→C         |
| 用術語承接術語   | 每個術語首次出現補一行功能描述 |
| 用比喻代替機制   | 先寫機制，再用比喻輔助         |

### 第四步：用卡片承擔術語，不讓正文變字典

降一級不等於每次都在正文長篇定義名詞。若術語會反覆影響判斷成本，就建卡或連既有卡；正文只保留當下推導需要的最小解釋。

例：`unit economics`、`switching cost`、`vertical SaaS` 這類詞，若每篇都重新解釋，正文會膨脹；卡片負責概念，文章負責事件推導。

## 沒這樣做的麻煩

### 讀者只能記名詞，不能複製判讀

高密度商業語言會讓文章看起來專業，但讀者若無法把術語放回因果鏈，只會記住「FDE 代表 SaaS 三支柱鬆動」這種結論句。下一次遇到不同市場事件時，判斷工具無法遷移。

### AI 改寫會保留原文讀者假設

AI 常把原文風格重寫得更流暢，但不會自動調整 reader contract。原文若寫給 VC / founder，改寫後仍可能保留高密度 shorthand，只是中文更順。

### 文章篇幅變短，但理解成本變高

跨領域文章最常見的錯覺是「短 = 清楚」。實際上，短句若省略中間因果，讀者需要自行補推導。對目標讀者而言，補推導的成本比多讀兩段更高。

## 判讀徵兆

| 訊號                                       | 該做的事                                     |
| ------------------------------------------ | -------------------------------------------- |
| 三句內連續出現三個以上商業術語             | 拆段、補首次解釋、或連到知識卡               |
| 一句話跨過三個以上因果步                   | 改成第一步、第二步、第三步的段落推進         |
| 文章讀起來像 VC memo 或 founder newsletter | 降到工程背景讀者可讀的 MBA 教材層            |
| 讀者能複述結論、但說不出判斷怎麼來         | 補中間機制、公式、數字或比較基準             |
| AI 改寫後文字變順、但術語密度沒下降        | 重新跑 reader-level pass，不把流暢度當可讀性 |

**核心原則**：跨領域分析的寫作目標是讓讀者複製判讀，不是讓原文變順。先定位原文與目標讀者的層級差，再調整術語密度與因果鏈步長。
