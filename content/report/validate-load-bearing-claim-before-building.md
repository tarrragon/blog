## 論述基礎與限制

本卡從一次教學模組的生產順序事故抽出：整合三個神經多樣性 skill 的方法論建立在一個承重論點上——「三者的硬衝突只有一組」。這個論點被相信、寫進 4 篇文章 + 1 個 skill + 索引，直到多輪審查的第三輪 steelman 才反證它錯（衝突不只一組），修正跨 6 個檔回改。限制：本卡談的是「有明確承重論點、且該論點驅動多個下游 artifact」的工作；一次性、下游少、或論點本身無關對錯（純風格）的工作不適用——不是每個假設都值得前置對抗驗證，只有承重的那個。

---

## 核心原則

**當一批工作建立在某個承重論點上，在動手蓋依賴它的下游 artifact 之前、先對抗性驗證那個論點。** 承重論點是「錯了就讓下游全部作廢或需要回改」的那個宣稱——方法論的核心主張、架構的核心假設、多處共用的 spec 或規則。它的錯誤不是局部的：它會等比例傳播到每一個建立在它之上的檔案。晚抓的代價跟「已經蓋了多少」成正比——先寫五篇再發現論點錯，要改五篇，而不是改一個論點。

所以驗證順序要跟依賴順序相反：**最被依賴的先驗**。承重論點在依賴圖的最上游，它該第一個被挑戰、且要通過才放行下游生產，而不是最後一個被審查。

---

## 反模式：先寫滿、再事後 steelman

自然的生產順序是「相信論點 → 據以寫滿 → 事後審查」。這個順序對承重論點特別危險，因為審查發生在下游都蓋好之後——steelman 這時才打掉核心，等於在一棟蓋好的樓上抽樑換柱。

兩個力量讓這個順序看起來合理：

- **產出感偏誤**：寫 / 蓋東西感覺像進度，回頭挑戰一個自己已經相信的論點感覺像繞路。於是先產出、後驗證。
- **同源相信盲區**：作者對自己的承重論點有結構性盲點——正因為相信它，才會拿它當地基。同源自審很難反證自己的地基（同 [#165](../register-violation-needs-cross-style-eyes/) 的機制：共享的直覺看不見自己的問題）。這代表承重論點的驗證不能靠自審、要靠對抗性 / 異源的挑戰。

事故實例：初版方法論寫「硬衝突只有一組」時，作者在列「不受影響的規則」時兩處都剛好漏掉會衝突的規則——不是分析不夠深，是**沒在動筆前先逐條枚舉挑戰這個論點**。等到 Round 3 steelman 逼著枚舉，才發現不只一組，但那時錯誤已經在 6 個檔裡。

---

## 修法：把承重論點的對抗驗證前置成生產的閘門

1. **動筆前先辨識承重論點**：問「這批東西建立在哪個宣稱上、那個宣稱錯了會讓多少下游作廢」。找出來的那個，就是要先驗的。
2. **對它跑一次對抗性挑戰、再蓋下游**：用一個「預設它錯、試圖反證」的姿態（獨立 / 異源 reviewer，或明確採對抗立場的自我步驟）逐條枚舉它的反例。承重論點的挑戰要前置成閘門——通過才放行下游生產。
3. **枚舉是那一步的核心、且沒有機械捷徑**：承重論點常是「只有一組 X」「所有 Y 都 Z」這類全稱或唯一性宣稱，反證靠逐條枚舉候選反例，跳過枚舉直接相信就是把「還沒找到反例」當成「不存在反例」（見 [#235](../integrate-conflicting-rulesets/) 招牌陷阱）。

這不是要對每個假設都前置 steelman——週邊、可局部修正的宣稱照常走事後審查流程就好。前置的只有承重的那一個，判準是「它錯的話下游要不要大改」。

---

## 跟審查時機的關係

多輪審查通常把 steelman 放在後段（frame 切換的最後一輪），這對抓多數問題是對的——大部分 finding 是下游的、事後抓也只改局部。但**承重論點是例外**：對它而言，後段 steelman 太晚，那個驗證該在生產前當閘門，不是在生產後當總結。換句話說，steelman 這個動作要用兩次——一次前置、只打承重論點、當放行閘門；一次後置、打全面、當收尾審查。把兩者混為一次（只在最後 steelman）就是本卡的事故。

## 判讀徵兆

| 訊號                                                         | 該做的事                                                                     |
| ------------------------------------------------------------ | ---------------------------------------------------------------------------- |
| 正要基於一個核心宣稱、開始寫 / 建多個依賴它的東西            | 先辨識這是不是承重論點（錯了下游要不要大改），是就先對它跑一次對抗挑戰再動筆 |
| 核心宣稱是「只有一組 X」「所有 Y 都 Z」這類全稱 / 唯一性主張 | 前置枚舉候選反例，別把「還沒找到反例」當「不存在反例」                       |
| 打算「先寫滿、最後一輪 steelman 再說」                       | 對承重論點這太晚 — steelman 用兩次：前置打地基當閘門、後置打全面當收尾       |
| 想自己審自己的核心假設                                       | 同源對地基有盲區（#165）；承重論點的挑戰交對抗 / 異源，不靠自審              |
| 事後審查跨多個檔改同一個核心錯誤                             | 這是「論點驗太晚」的症狀；下次把那類論點的驗證前移到生產前                   |
