## 結論

文章中的 meta 資訊（「這篇文章的目的是…」「本文邊界是…」「先交代脈絡，否則…」）服務的是作者，不是讀者。讀者打開文章的問題是「這篇文章對我有什麼用」，而非「作者為什麼寫這篇文章」。

## 案例

### 案例一：整理目的 blockquote

| 版本   | 內容                                                                                 |
| ------ | ------------------------------------------------------------------------------------ |
| 修正前 | `> **整理目的**：這不是一次性事件的檢討報告，而是對「工具設計者的責任邊界」的反思。` |
| 修正後 | （整段刪除，改為直述讀者能帶走什麼）                                                 |

「整理目的」告訴讀者作者為什麼寫這篇文章。但讀者的問題不是「你為什麼寫」，而是「我為什麼要讀」。兩者的差別：

- 作者視角：「這是一篇反思，不是事件報告」→ 在做分類
- 讀者視角：「讀完這篇我能改善什麼？」→ 在評估投入回報

### 案例二：脈絡解釋

| 版本   | 內容                                       |
| ------ | ------------------------------------------ |
| 修正前 | 「先交代脈絡，否則後面的事件沒有參照點。」 |
| 修正後 | （整句刪除，直接進入脈絡敘述）             |

這句話是寫給自己的編輯註記——「提醒自己為什麼要寫背景段」。讀者不需要知道作者的編輯考量。章節標題（「背景：我們怎麼管理版本和工作項目」）已經告訴讀者接下來是脈絡交代。

## 判斷標準

寫完一段 meta 描述後問：**這段話消失後，讀者的閱讀體驗會變差嗎？**

- 「整理目的」消失 → 讀者仍然能從內文判斷這是反思而非報告 → 不會變差 → 刪除
- 「先交代脈絡」消失 → 讀者仍然會讀到脈絡段 → 不會變差 → 刪除
- semver 背景段消失 → 讀者會在事件段卡住 → 會變差 → 保留

meta 資訊的本質是「替讀者做他已經能自己做的事」。讀者能從標題和內文推斷文章類型，不需要作者顯式宣告。

## AI 寫作的傾向

AI 生成的文章高頻出現 meta 資訊，因為 AI 的生成過程包含「規劃→組織→寫作」三步，meta 資訊是規劃步驟的殘留——AI 把內部的推理過程（「我接下來要先交代脈絡」）外露到了文章中。

生成端防護：完成初稿後掃描所有 blockquote 和段首句，問「這句在描述內容還是在描述寫作過程？」描述寫作過程的句子刪除。

**場景邊界**：此判斷適用於短篇分享文。長篇技術文件或 RFC 的 scope 聲明（「本文不討論 X」）有不同作用——幫讀者快速判斷是否繼續讀，屬合理 meta。
