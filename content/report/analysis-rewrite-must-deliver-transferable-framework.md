## 核心原則

外部分析改寫的交付物是可遷移框架、不是風格轉換。把其他分析師文章交給 AI 改寫時，任務目標不能停在「改成本站語氣」「更正向」「更像教學」；真正要交付的是讀者能帶到下一個市場事件使用的判讀工具。

| 改寫層級 | 產物                                 | 失敗模式                 |
| -------- | ------------------------------------ | ------------------------ |
| 風格轉換 | 同一篇文章換語氣、換標題、換段落     | 像摘要或二次評論         |
| 結構轉換 | 把 WRAP process 改成教學章節         | 可能仍偏離標題承諾       |
| 框架轉換 | 事件訊號、機制、風險、預警、遷移用法 | 讀者能用來分析下一個事件 |

判別問題：「讀者讀完後，是否多了一個可重用的判斷問題或檢查表？」沒有，就還只是改寫。

---

## 情境

`content/business/` 的建立過程是一場從「外部分析文章」到「本站教學型商業知識」的轉換實驗。commit 演變顯示，早期版本雖然有 WRAP、知識卡與 case analyses，但仍多次被讀者 feedback 拉回核心問題：

- `#140`：Widen Options 不能用稻草人修辭展示作者結論。
- `#141`：WRAP 是內部工具，不能直接當文章章節。
- `#142`：即使章節標題改了，正文仍要對齊標題承諾。
- `writing-down-a-level`：目標讀者是工程背景讀者，不是原文的投資人或創辦人。

這些問題共同指向一件事：AI 轉換文章風格不夠。文章要從「原作者如何看這件事」轉成「本站讀者以後如何判讀同類事件」。

## 理想做法

### 第一步：把任務定義成 frame extraction

輸入外部文章時，先問：

| 問題                                    | 作用                       |
| --------------------------------------- | -------------------------- |
| 這篇原文用什麼 frame 看事件？           | 辨識原作者判讀，不直接繼承 |
| 這個 frame 對本站讀者有沒有可遷移價值？ | 決定是否值得寫成文章       |
| 若要遷移，讀者下次要看哪些訊號？        | 把評論轉成判讀工具         |
| 哪些術語需要卡片支撐？                  | 避免正文變成術語堆疊       |

若事件只能產出「某公司發生某事」的摘要、沒有可遷移框架，就放在筆記，不硬寫成 case-analysis。

### 第二步：正文結構服務可遷移框架

教學型商業分析的穩定結構可以是：

```text
事件本身 → 結構性機制 → 長期影響 → 預警訊號 → 可遷移框架
```

這是責任順序，可依標題承諾調整。事件段讓讀者進場；機制段教為什麼；長期影響處理時間軸；預警訊號讓框架可被反證；可遷移框架讓讀者能帶走。

若某篇文章的標題承諾需要不同順序，可以調整；但最後仍要回答「這個事件教讀者下次看什麼」。

### 第三步：把原文觀點轉成判斷問題

風格轉換會把原文句子改得更順；框架轉換會把原文觀點改成讀者可問的問題。

| 原文觀點型句子                  | 框架型問題                                               |
| ------------------------------- | -------------------------------------------------------- |
| AI legal tools 會擠壓知識工作者 | 這個工具在擠壓應用層、新創、工作者哪一層？               |
| FDE 是資料平台的新戰場          | 它是否鬆動 SaaS 的 distribution、workflow、data 三支柱？ |
| 收購代表基礎設施整併            | 這是純收購、賽道整併，還是算力廠商垂直整合？             |

判斷問題比結論更有價值，因為問題可以帶到下一個 case。

### 第四步：用預警訊號保護框架

可遷移框架需要知道何時失效。每篇 case-analysis 至少要有一段預警訊號，列出哪些觀察會推翻或削弱本文判讀。

沒有預警訊號的框架只是一組漂亮分類。讀者需要知道「何時重新評估」，才會把框架當工具，而不是把它當口號。

## 沒這樣做的麻煩

### 文章會變成原文摘要

AI 很擅長把原文改成不同語氣，但摘要仍然沿著原文的 frame 走。讀者得到的是「另一個版本的原文」，不是本站累積出的知識單元。

### 事件評論不可重用

若文章只回答「這件事代表什麼」，下次遇到不同公司、不同市場、不同產品時，讀者還是要從頭判斷。可遷移框架要回答「下次遇到相似事件，我要看哪些訊號」。

### 寫作規範會被誤解成表面風格

正向陳述、核心原則先行、商業邏輯先於 CASE 不是語氣規則；它們是把思考過程變成可重用知識的結構規則。若只做風格轉換，會通過部分表面檢查，但仍不符合知識庫目標。

## 判讀徵兆

| 訊號                           | 該做的事                                     |
| ------------------------------ | -------------------------------------------- |
| Prompt 只說「改成我們的風格」  | 改成「抽出可遷移判讀框架」                   |
| 文章結尾只有事件結論           | 補「下次遇到同類事件要看什麼」               |
| 讀者讀完只能說出某公司發生什麼 | 補訊號、機制、風險、預警與路由               |
| 文章沒有預警訊號或 Tripwire    | 補失效條件，讓框架可被反證                   |
| 文章大量沿用原文 frame         | 回到 source layering，區分原文判讀與本文推導 |

**核心原則**：外部分析改寫的交付物是讀者可遷移的判讀框架，把原文變順只是表層。沒有框架、預警與下一步路由，文章仍停在摘要層。
