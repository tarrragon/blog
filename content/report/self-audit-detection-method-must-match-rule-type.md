## 論述基礎與限制

本卡基於一次多輪審查的 self-application sweep 事故。用 grep 對一批 session 內容掃一組自訂規範（#229 分析開頭不講創辦敘事、#230 寫作動機不入正文、負向起手、喊話、裝飾符號、誇飾），回報「近乎 clean」。一個異源 reviewer 隨後 eyeball 抓到一篇文章的開頭正是 #229 的敘事違反——整段併購史敘事。根因：grep sweep 對 #230（有穩定句型「我們為什麼」）有效、對 #229（敘事開頭沒有關鍵詞）結構性失明，卻把兩條都算進「clean」。限制：案例來自寫作審查，但原則適用於任何「用單一偵測方法對一組異質規則做批次自審」的情境（lint 掃程式碼規範、grep 掃合規字串都同構）。

---

## 核心原則

**一個自審對某條規則回報 clean，只有在偵測方法看得見該規則的違反形態時才可信。** 規則依「違反有沒有穩定的可偵測標記」分兩類：有標記的（動機框架的「我們為什麼」句型、裝飾符號、負向單詞）grep 抓得到；無標記的（敘事開頭、結構失配、register 失調、論證斷點）沒有關鍵詞可掃、grep 結構性看不見。用一輪 grep 掃完一組混合規則就回報全部 clean，等於把「這條沒違規」跟「這條我的方法看不到」混為一談。

自審前的必做步驟是把規則按可偵測性分兩堆：可 grep 的用 grep 掃、不可 grep 的標記為「本方法未涵蓋、需 eyeball 或異源複核」。回報時兩種訊號分開講——「grep 掃過、clean」跟「本方法偵測不到、未複核」不是同一件事，不能都塞進 clean。

---

## 反模式：把偵測失明當成沒違規

用 grep sweep 掃一組規則、逐條回報 clean，包含那些 grep 本來就抓不到的規則。這個 clean 是假 clean——不是內容乾淨、是方法看不見。危險在於假 clean 讀起來跟真 clean 一模一樣，且自審者在「我掃過了」的錯覺下停止追查，把偵測方法的盲區當成內容的乾淨。

上述事故裡有兩層失敗疊在同一個漏網上：編輯該檔時範圍界定過窄、沒把已確立的 #229 套到整檔開頭（只改被點名的段落）；接著自審的 grep 對 #229 這類無關鍵詞規則失明、回報 clean。兩層都靠異源 reviewer 的 eyeball 補回——這正是無標記規則的真防線不在自審方法內、而在文體異源視角。

## 判讀徵兆

| 訊號                                                        | 該做的事                                                          |
| ----------------------------------------------------------- | ----------------------------------------------------------------- |
| 用一輪 grep sweep 就回報一整組規則「全部 clean」            | 先把規則按可偵測性分堆、無關鍵詞的標「未涵蓋」、不併入 clean      |
| 某條規則的違反沒有穩定句型 / 關鍵詞（敘事、結構、register） | grep 對它結構性失明、改用 eyeball 或異源 reviewer、回報時標明     |
| 自審回報 clean 後就停止追查                                 | 問「這個 clean 是沒違規、還是我的方法看不見」再決定要不要異源複核 |
| 編輯一個早於某條 session 規則的檔案、只改被點名的段落       | 已確立的規則適用整檔、不只正在動的段落——把規則掃過整檔一次        |
