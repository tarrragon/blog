## 論述基礎與限制

本卡的論述基於 **1 個 batch** 的漏抓事故：電子業採購 planning 模組（6 篇 + 13 卡）經三輪 multi-round review（Round 1 compliance / Round 2 cadence + 讀者旅程 / Round 3 steelman + outbound）修訂後 merge，使用者仍判定整個模組是「講故事不是商業分析教學」——三輪、九個以上 reviewer frame，沒有任何一個 finding 指向知識類型錯位。具體限制：

- **抽樣密度是經驗值**：每篇抽 3-5 個承擔判準的核心斷言是本卡給的起點，未經多 batch 校準。
- **支撐類型的分類法未窮舉**：機制推導 / 量化 / 可驗來源 / 實機驗證是目前列出的合法支撐形態，不同內容類型可能有其他形態。
- **單一事故來源**：genre 失配在其他方向（分析文章寫進操作分類、教材寫成論文）的對稱情境未驗證。

## 核心原則

**多輪審查的 frame 集合要包含「斷言支撐」frame：抽樣每篇承擔判準的核心斷言，問「這個斷言靠什麼成立——機制推導、量化、可驗來源，還是口吻與權威」；並在模組層對照「這個模組的知識類型（分析 / 敘事 / 操作）跟它所在分類的定位與 sibling 的 house style 一致嗎」。** 既有的審查 frame——字句層 compliance、cadence、讀者旅程、steelman——都在文字表面與結構層操作。一個模組可以每一句都合規、每個結構都到位，整體卻是錯的知識類型：結論都在、支撐全缺的經驗談，穿著教學結構的衣服全數通過。

這是 frame 覆蓋的 design gap，不是 execution gap：三輪都確實跑了、reviewer 都盡責、grep 都命中該命中的，但 frame 清單裡沒有這個維度——加輪數、加 keyword 都補不上，要補的是一個新 frame。

斷言支撐 frame 的兩層操作：

| 層級   | 檢查問題                                                            | 失配訊號                                       |
| ------ | ------------------------------------------------------------------- | ---------------------------------------------- |
| 斷言層 | 抽 3-5 個承擔判準的斷言，各自的支撐是機制 / 量化 / 來源、還是口吻？ | 支撐是「厲害的人都這樣」「經驗上如此」         |
| 模組層 | 模組的知識類型跟分類定位、sibling house style 一致嗎？              | 分類的分析語言（算式、結構分析）模組完全用不上 |

## 具體 case

### case 1：steelman 在句子層放行了判準層的缺口

Round 3-B steelman reviewer 的檢查項包含「數字 / 閾值有無源頭」——它在**句子層**操作：「單一家不超過**約**七成」有標示約數、語氣誠實，放行。斷言支撐 frame 問的是**判準層**的問題：「七成的推導在哪？讀者換情境能重算嗎？」同一句話、兩個 frame、兩種結論——說明斷言支撐不是 steelman 的子集，是獨立維度。

### case 2：模組層的 genre 失配沒有任何 frame 負責

該模組放在商業分析分類下。分類的 house style 由既有內容錨定：案例拆解文章有算式（GPU 閒置成本估算）、有市場結構分析（整併週期、通路格局）。新模組進來時，三輪裡沒有任何 frame 把模組跟 sibling 的知識類型做對照——cross-surface 檢查（title 對齊、跨章一致）都在模組**內部**操作，「模組作為整體跟分類定位是否同一種知識」無人負責。

### case 3：全部通過本身就是訊號

三輪審查產出的 finding 全部是字句與結構層（負向陳述、cadence 同骨、broken link、enumeration 缺項）。回頭看，「沒有任何一個 finding 質疑斷言站不站得住」這件事本身就該觸發警覺——finding 類型的分佈反映 frame 集合的覆蓋範圍，全部集中在表面層代表深層無人在看。

## 沒這樣做的麻煩

### 審查給出假陰性、返工成本最大化

知識類型錯位是模組級問題，修法是重寫不是修句。審查在字句層全過等於給了「品質達標」的錯誤信號，錯位一路活到 merge 之後才被使用者抓到——此時六篇文章的字句層已經打磨過三輪，打磨的功夫全部作廢。錯位越早抓（理想是第一篇成稿時），返工越小。

### 審查框架的信任被侵蝕

「三輪全過的東西被判定粗糙」讓審查制度本身失去公信力。防護是誠實標示覆蓋範圍：frame 集合覆蓋了哪些層、哪些層不在本輪範圍，讓「全過」的語意精確化為「已覆蓋層全過」。

### 同類模組會繼續漏

經驗談轉教學是可重複的生產路徑（訪談、社群素材、口述整理都會再進來）。frame 缺口存在一天，每一批同類素材都會以同樣方式通過審查。

## 跟其他抽象層原則的關係

- **[#126 寫作 review 是多軸完整性、不是單軸深度](../writing-review-multi-axis-completeness/)**：本卡是 #126 的一次實證——七軸（frame / instance / surface / scope / cadence / timing / granularity）都有跑，仍 systematic miss，因為「斷言支撐」不在七軸展開的 frame 清單裡。多軸完整性的軸清單本身需要隨漏抓事故擴充，本卡補的是 frame 軸底下的「知識類型」維度。
- **[#153 漏抓先分 design gap vs execution gap](../review-miss-diagnose-design-vs-execution-gap/)**：本卡是 design gap 的典型案例——三輪執行完整、缺口在 frame 清單設計。依 #153 的判別，修法是改框架（加 frame），加輪數或加 keyword 是無效方向。
- **[審查缺 outside-in 讀者 frame](../review-lacks-outside-in-reader-frames/)**：sibling、同屬「inside-out 表面審查的結構盲區」系列。outside-in 系列補的是「讀者拿內容去做什麼」的用途盲區；本卡補的是「內容的斷言憑什麼成立」的支撐盲區。兩者都是從文字表面看不到、要換座標系才可見的維度。
- **[#148 跨輪 review 停止訊號](../cross-round-review-stopping-signal/)**：#148 說停止判讀看「想不出新 frame」。本卡給這個判讀補一個具體檢查：想新 frame 時，把「finding 類型分佈」攤開——若全部集中在表面層，斷言支撐 / 知識類型層就是還沒動用的 frame 方向。

## 判讀徵兆

| 徵兆                                                     | 該做的行動                                                  |
| -------------------------------------------------------- | ----------------------------------------------------------- |
| 多輪全過的內容被使用者判定「粗糙 / 像故事 / 深度不足」   | frame 覆蓋缺維度（design gap）——補斷言支撐 frame、非加輪數  |
| 歷輪 finding 全部是字句與結構層                          | 深層無人在看的訊號——下一輪排斷言支撐 frame                  |
| 模組 description 自稱「策略 / 分析」、內文支撐是權威口吻 | 斷言層失配——逐篇抽核心斷言檢查支撐類型                      |
| 同分類 sibling 有算式與結構分析、新模組全篇敘事          | 模組層 genre 失配——對照分類 house style 重估知識類型        |
| 素材來源是經驗談 / 訪談 / 口述                           | 高風險 batch——斷言支撐 frame 排進第一輪、錯位要在成稿早期抓 |

## 適用範圍與邊界

- **適用**：教學模組、分析文章、規範文件的多輪審查——特別是素材來源為經驗談、且目的地分類有成熟分析語言時。
- **邊界**：
  - **抽樣不是逐句**：每篇抽 3-5 個承擔判準的斷言即可，逐句要 citation 會把審查變成論文評審。
  - **支撐類型依知識目標判定**：操作型文章的合法支撐是實機驗證、分析型文章是機制與量化、案例型文章是可回溯來源——frame 檢查的是「有沒有支撐」與「類型跟文章的知識目標對不對」，不是統一要求某一種形態。
  - **心態 / 生態類內容不適用斷言支撐檢查**：這類內容的價值本來就是經驗分享，正確的處理是標明知識類型並跟分析內容分離（路由問題），不是要求它長出機制。

## Self-case：本卡的觸發來源

觸發於採購 planning 模組的漏抓事故：模組經三輪 multi-round review 修訂（Round 1 三個 reviewer、Round 2 / Round 3 各數個 frame），merge 後使用者判定「不是要用講故事的方法做教學，需要的是商業分析角度的教學文章」。

回查三輪 finding：全部落在字句層（負向陳述、口語）、結構層（cadence、編號、連結）、讀者旅程層（術語卡覆蓋、入口動機）——沒有任何 finding 質疑「代價不對稱」「不超過七成」這些核心判準的支撐。steelman reviewer 檢查過「閾值有無源頭」但在句子層放行（有標約數即過）。模組層面，商業分類的 house style（案例拆解有算式與市場結構分析）跟新模組的純敘事落差，三輪裡無任何 frame 負責對照。素材端的根因（經驗談轉換缺機制重建）另立 [#216](../anecdotal-source-needs-mechanism-reconstruction/)，本卡記審查端：這類錯位要有專屬 frame 才看得見。

**Update（2026-07-10）**：本卡立卡後，multi-round-review skill 已新增 Round 1-E 斷言支撐 reviewer；採購模組重寫時實際執行——Round 3 的斷言支撐 self-sweep 與 steelman 兩個獨立 reviewer 收斂抓到同一個跨篇判準矛盾（免費選擇權 vs 訂單信用配給），驗證了 frame 的增量價值。卡內的漏抓描述以重寫前的三輪審查為準。
