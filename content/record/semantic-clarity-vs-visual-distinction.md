## 問題情境

在技術文章中使用圖例說明兩個相反概念：

```markdown
> 🟢 HIGHLIGHT = 接收檔案變更的目標 commit（A）
> 🔴 REVERSE = 含有不該屬於它的檔案變更的 commit（C）
```

這個做法看起來用 emoji 區分了兩個概念，但實際上有根本缺陷。

<!--more-->

## 問題的層次

### 層次 1：視覺層（最淺）

emoji 在不同容器寬度下渲染不穩定 → 可能導致斷行錯位

```text
看到：🔴 
     REVERSE = ...
```

**但這不是真正的問題。** 這只是表面症狀。

### 層次 2：語意層（更深）

HIGHLIGHT 和 REVERSE 代表**兩個完全不同的概念**：

- **HIGHLIGHT**：作為目標接收變更的 commit
- **REVERSE**：作為變更來源卻含有不該屬於它的內容的 commit

既然語意不同，用「同一行 + emoji 區分」無法充分表達它們的區別。

### 層次 3：結構層（根本）

在 `> 引用區塊` 的語境下：

```markdown
> 🟢 HIGHLIGHT = ...
> 🔴 REVERSE = ...
```

用 emoji 作為**唯一的視覺區分**是不夠的，因為：

1. emoji 可能不渲染或渲染異常（在某些終端、編輯器、瀏覽器）
2. emoji 對螢幕閱讀器用戶無法被正確解讀
3. emoji 無法作為可靠的**語意標記**

引用區塊本身就暗示「這是附加說明」，而兩個同等重要的概念（HIGHLIGHT vs REVERSE）不應該都淪為「附加說明」。

## 正確的做法

既然是不同的語意單位，應該分開表達：

### 做法 A：結構化列表

```markdown
**四個 commit 的角色**：

- **A**（接收目標）：commit C 中對檔案的修訂應該屬於這裡
- **C**（變更來源）：同時改了目標檔案和其他 6 個檔案
```

每個概念佔據獨立的列表項，語意清晰且結構穩健。

### 做法 B：分段論述

```markdown
## 目標 commit（接收變更）

commit A 是…

## 變更來源 commit（含不該屬於它的檔案）

commit C 是…
```

用小節清晰地分隔兩個概念，讀者無需依賴 emoji 理解區別。

## 檢查清單

寫完說明後問自己：

1. **語意層**：這裡有幾個完全不同的概念？
   - 若 ≥ 2，不應該用「同一行 + 視覺標記」表達
2. **結構層**：用什麼方式承載這些概念？
   - 引用區塊？列表？小節？
   - 該方式能否可靠地表達差異？
3. **可靠性**：去掉視覺標記後，讀者還能理解嗎？
   - 若不能，代表依賴了不可靠的視覺標記

---

**教訓**：不能用「視覺區分」替代「結構清晰」。視覺可能失效，結構才是基礎。
