DevOps 是把軟體從「寫完」帶到「在 production 穩定活著」的整套文化與實務，涵蓋一條完整的交付生命週期：**先把地基鋪好（Infra），再讓每次變更能被驗證、建置、交付（CI/CD），最後讓上線的服務在運行期扛得住負載、掛得了會恢復、帳單不失控（運行期維運）**。三個階段各自是獨立主題、串起來才是 DevOps。本站把這條生命週期的每一階各自展開成獨立的成熟度階梯，這頁是三階段的全景入口與路由。

## 三個階段

| 階段 | 系列                          | 責任                                                                                     | 什麼情境進來                                              |
| ---- | ----------------------------- | ---------------------------------------------------------------------------------------- | --------------------------------------------------------- |
| 地基 | [Infra 基礎設施建置](/infra/) | 服務跑起來**之前**必須存在的地基：IaC、身分憑證、網路、環境分離、核心服務、治理          | 要從零把雲端地基一階一階做起來、或接手 / 稽核既有環境     |
| 管線 | [CI/CD 教學](/ci/)            | 變更**如何**被驗證、建置、交付：gate、artifact、rollout、rollback，跨不同部署場域        | 要讓每次變更能被穩定驗證與交付、workflow 失敗要判讀下一步 |
| 維運 | [運行期維運](/operations/)    | 設計好的服務**在 production 怎麼活下來**：負載、擴展、流量管控、探活、容量、高可用、成本 | 服務上線後要扛負載、防過載、規劃容量、控成本              |

三者是接續的：Infra 鋪好的地基，是 CI/CD 管線部署的目標環境；CI/CD 管線的產出（artifact），是運維要讓它在 production 活下來的對象。而 Infra 自己的變更走 PR 時，用的也是 CI/CD 管線——三階段彼此咬合，不是各走各的。

## 為什麼拆成三個獨立系列、而不是一個大目錄

每一階都深到需要自己的成熟度階梯（各有 00-09 模組與知識卡），硬塞進同一個目錄會壓垮各自清楚的責任邊界。拆開讓讀者能從自己當下的問題直接落地——正在鋪網路地基的人不必先滑過一堆容量規劃，正在救火調流量的人不必先讀完 IaC。這頁負責把三階段的關係講清楚、給跨階段的路由；深度內容在各系列裡。

> 術語說明：本站把 **「運行期維運」** 獨立成一個系列（`/operations/`），涵蓋負載 / 擴展 / 容量 / 成本這些**運行期**議題；而 **「DevOps」** 在本站指的是這整條橫跨 Infra + CI/CD + 運維 的交付生命週期文化——也就是這一頁。
