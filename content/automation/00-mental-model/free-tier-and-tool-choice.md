免費膠水層能不能撐住你的量，要看的限制不是「一天總共幾次」，而是「同一瞬間能有幾個請求在跑」。這兩者常被搞混，導致人用錯誤的方式估算容量。個人 blog 的流量幾乎不可能打爆總量限制，卻可能在某篇文章被分享的那一刻，短時間湧入的併發請求撞上併發上限。先建立正確的思考單位，才知道免費夠不夠、什麼時候要換工具。

## 免費額度該看併發，不看總量

Google Apps Script 對個人（gmail.com）帳號的關鍵限制有這幾條：

| 限制             | 個人帳號額度   | 對流量 beacon 的意義                             |
| ---------------- | -------------- | ------------------------------------------------ |
| 單次執行時間     | 6 分鐘 / 次    | 一次寫一筆瀏覽遠遠用不到，接住就寫、幾百毫秒結束 |
| 同時併發執行     | 30 / 使用者    | 真正的天花板：同一瞬間最多 30 個 beacon 在處理   |
| 觸發器每日總時間 | 90 分鐘 / 天   | 影響的是排程彙總（模組四），不是接收 beacon      |
| 對外 URL Fetch   | 20,000 次 / 天 | 膠水層主動打外部 API 才算，接收 beacon 不算      |

對一個接住 beacon、寫一筆進 Sheet 的膠水層，binding 的限制是**同時併發 30**。這條線的意思是：只要不是在同一瞬間有超過 30 個人的瀏覽器同時打 beacon 進來，就不會撞牆。一篇文章一天被看一千次、但這一千次分散在整天，任何瞬間的併發都遠低於 30，完全沒問題。會出事的情境是「某篇文章上了熱門、一分鐘湧入幾百個瀏覽」——這時併發才可能逼近 30，多出來的請求會拿到錯誤。

值得注意的是，Google 並沒有為個人帳號的 web app **每日呼叫次數**公布一個硬性數字，所以正確的估算方式不是去湊「一天幾次」，而是問「我的流量會不會在某個瞬間有 30 個以上的同時請求」。對個人 blog，答案幾乎都是不會。

## Apps Script vs Cloudflare Workers 的適用邊界

Apps Script 跟 Cloudflare Workers 都能當免費膠水層接住 beacon，但它們補的場景不一樣。選錯不會壞，但會讓你在錯的地方費力。

**Apps Script 適合資料量小、需要人直接讀寫試算表的場景。** 它最大的優勢是 Google Sheet 同時是儲存體跟儀表板——資料一 append 進去，你打開試算表就能看、能排序、能畫圖、能用樞紐分析，不必另外做前端。起步不需要信用卡、不需要買 domain、不需要學部署工具。代價是效能：每次執行有冷啟動延遲，Sheets 當資料庫在資料列數很多時讀寫會變慢，併發上限只有 30。對「一個人的 blog 想知道哪篇有人看」，這些代價都碰不到。

**Cloudflare Workers 適合量體較大、需要低延遲的場景。** 免費方案給到每天十萬次請求量級，冷啟動幾乎感覺不到，全球邊緣節點讓 beacon 延遲很低。代價是它沒有內建的試算表 UI——資料要存進搭配的 KV 或 D1（免費的 SQLite），看資料得自己寫查詢或做一個小前端。它更接近「一段真的後端程式」，彈性大但要自己搭的部分多。

選型判準用一句話收斂：**先問資料要不要讓人打開試算表直接看、量會不會大到讓 Sheets 變慢。** 想要試算表即儀表板、量不大——Apps Script。要低延遲或預期量大、能接受自己搭查詢介面——Workers。本指南先走 Apps Script，因為它把「看資料」這件事直接解決了，最適合第一次做流量統計、想快點看到成果的人。模組五會給出「Sheets 開始撐不住時怎麼判斷、怎麼往 Workers 遷移」的訊號與路徑。
