# Codex 工作規範（blog 專案）

本文件為 Codex 在 `blog/` 專案撰寫技術教材時的行為準則。採**正面表述**原則：以「應該這樣做」替代「禁止那樣做」，並在每段產出後執行一次**負向表述檢查循環**，確保文字保持建設性語氣。

---

## 1. 專案範圍

| 系列目錄 | 定位 |
|---------|------|
| `content/backend` | 後端通用知識庫（服務選型、資料庫、快取、佇列、觀測、部署、可靠性、安全、事故應對） |
| `content/go` | Go 語言入門到中階教材 |
| `content/go-advanced` | Go 進階主題（並行、效能、架構模式） |
| `content/python` / `content/python-advanced` | 既有參考風格來源 |

**當前任務**：以 `../server/` 下的 Go WebSocket 伺服器為素材，轉化為 `content/go` 與 `content/go-advanced` 的教材。

**素材要點**：

- `main.go`：啟動流程、依賴注入、HTTP 路由、優雅關閉
- `websocket_server.go`：WebSocket 升級、client 生命週期、讀寫 pump
- `event_dispatcher.go`：channel fan-in、事件去重、狀態更新
- `session_registry.go`：基於 `sync.RWMutex` 的並行狀態管理
- `file_watcher.go`：檔案系統監控與 JSONL 事件讀取
- `hooks_handler.go`：HTTP JSON handler 與非阻塞 channel 發送
- `*_test.go`：table test、注入時鐘、並行驗證

## 2. 寫作規範（以 SKILL 為準）

**權威來源**：`book_overview_v1/.claude/skills/compositional-writing/SKILL.md`

每次開始撰寫前，**先讀取該 SKILL 全文**，並依據下方觸發路由讀取對應 reference（最多一層深）。本章節僅為摘要，具體檢查項以 SKILL 為準。

### 2.1 三大支柱

| 支柱 | 要求 |
|------|------|
| Atomization 原子化 | 一篇文章聚焦一個概念，能獨立閱讀與重用 |
| Explicit Intent 意圖顯性 | 第一段就交代「為什麼要讀」與「讀完能做什麼」 |
| Searchability 可查詢性 | 關鍵字前置、標題 grep 友善、術語保持一致 |

### 2.2 五大原則（摘要）

1. **原子化**：每篇文章 / 每個段落只承載一個概念；讀者同時需要記住 7 個以上概念即應拆分。
2. **索引建立**：系列入口（`_index.md`）只做路由，不承載細節；引用深度最多一層。
3. **意圖顯性與商業邏輯**：寫「為什麼這樣設計」「要達成什麼目標」，再進入程式碼細節。業務邏輯先於具體程式碼範例。
4. **可查詢性**：標題含可 grep 的術語；表格使用 `:`、`|`、`→` 等穩定分隔符。
5. **欄位設計**：同一文件的不同章節從不同角度切入，保持互補而非重複。

### 2.3 長篇技術文章的推理路徑

長章節保留完整思路：**觀察 → 詮釋 → 策略 → 執行**。先呈現業務情境與取捨，再進入程式碼。

### 2.4 觸發路由

| 情境 | 讀哪份 reference |
|------|----------------|
| 寫或改程式碼註解 | `compositional-writing/references/writing-code-comments.md` |
| 起草 / 改寫文件 | `references/writing-documents.md` |
| 撰寫完整長篇技術文章 | `references/writing-articles.md` |
| 設計欄位 / frontmatter | `references/designing-fields.md` |
| 驗證寫作品質 | `references/meta-metrics.md` |

### 2.5 Go 教材專屬補充

- 以業務情境切入（「為什麼需要事件去重？」），再引出 Go 的 channel / mutex / goroutine 做法。
- 範例先展示成熟 Go 專案的慣用寫法，再回到 `server/` 的具體實作。
- 保留 `server/` 的設計取捨（如非阻塞 channel send 的理由），轉化為通用教學結論，保持中立敘事。

## 3. 決策與分析規範（以 WRAP SKILL 為準）

**權威來源**：`book_overview_v1/.claude/skills/wrap-decision/SKILL.md`

### 3.1 用戶簡寫對應

| 用戶輸入 | 實際動作 |
|---------|---------|
| 「分析」「warp」「思考決策」 | 執行 WRAP 流程 |
| 「做分析」「查資料再分析」「評估方案」「提案評估」「架構決策」 | 執行 WRAP 流程 |

### 3.2 WRAP 流程骨架

1. **Anchor Check**：確認客戶、當前目標、議題是否影響決策品質或開發效率
2. **Step 0 資料充足度閘門**：確認證據足夠，辨識假設替代資料的情況
3. **W — Widen Options**：擴增選項（含「零工具 / 既有機制改造 / 新增工具」三類涵蓋）
4. **R — Reality Test**：以原始來源核對、基本率優先、逐項對照清單
5. **A — Attain Distance**：揭露機會成本、10/10/10、核心優先事項對齊
6. **P — Prepare to Be Wrong**：Premortem、失敗模式、安全係數、回退計畫、絆腳索

具體檢查項（如 Tool selection 4 問、Consider the Opposite、悖論識別清單）以 SKILL 全文為準。

### 3.3 研究階段的網路查詢

優先使用**原始來源**：官方文件、專案 repository、設計文件、原始碼、release notes。對工程範例優先選用成熟 Go 專案。

採四輪迭代結構：

| 輪次 | 焦點 |
|-----|------|
| 發散搜尋 | 跨多個相關領域的廣泛關鍵字 |
| 具體搜尋 | 具體專案範例與實際程式碼結構 |
| 精準搜尋 | 高品質來源細節、文件、精確術語 |
| 反向搜尋 | 批評、反例、限制、失效情境 |

每輪結束後揭露當前偏誤，說明「什麼證據能改變結論」。清單類答案必須逐項對來源核對，單一搜尋的結果保留為候選假設。

### 3.4 教材情境的應用要點

- 用 WRAP 避免課程結構過度貼合 `server/` 專案。
- W 階段將範例擴展到多個成熟 Go 專案。
- R 階段以真實 Go 程式碼庫驗證架構主張。
- 將研究成果轉化為**中立教學章節**，保持普遍適用性，作為通用教材而非專案維護指示。

## 4. 正面表述原則

本專案以**建設性語氣**為預設風格。描述規則時，聚焦在「應該怎麼做、會得到什麼效果」；描述邊界時，以**範圍界定**取代**否定陳述**。

### 4.1 詞彙替換建議表

| 負向表述 | 建議替換 |
|---------|---------|
| 「不是 X」 | 「應以 Y 為準」 / 「屬於 Y 範疇」 |
| 「不可以 X」 | 「請改用 Y」 / 「建議以 Y 取代」 |
| 「不可能 X」 | 「X 的成立條件是 Z，缺少 Z 時以 Y 處理」 |
| 「禁止 X」 | 「採用 Y 作為標準做法」 |
| 「避免 X」 | 「優先採用 Y」 |
| 「X 是錯的」 | 「Y 更貼合這個情境，理由是 Z」 |
| 「不要用 X」 | 「使用 Y 可以得到 Z 效果」 |

### 4.2 正面表述的結構範本

每段規範性描述採下列三段結構：

```
原則：[以動詞開頭陳述該做什麼]
理由：[為什麼這樣做會更好]
範例：[示範符合原則的寫法]
```

### 4.3 保留否定詞的情境

有三類情境仍然保留精確的否定描述，以維持技術正確性：

1. **物理限制陳述**：例如「channel 關閉後重複關閉會 panic」——這是 Go 語言事實，屬於技術描述而非規範語氣。
2. **安全警示**：例如「併發寫入同一 map 會觸發資料競爭」——讓讀者能識別風險訊號。
3. **反模式對照**：在「正例 / 反例」對照表中，反例欄位可保留具體的錯誤寫法，讓讀者辨識差異。

在以上三類情境外，預設採正面表述。

## 5. 生成後檢查循環（每段必做）

每寫完一段內容（一個章節或一個主題段落），執行以下檢查循環。**循環通過後才能寫下一段**。

### 5.1 檢查項目

| 編號 | 檢查項 | 通過條件 |
|-----|-------|---------|
| C1 | 意圖顯性 | 段落第一句能獨立說明「這段在講什麼」 |
| C2 | 原子化 | 讀者只需追蹤 ≤ 7 個概念就能讀懂 |
| C3 | 業務邏輯前置 | 程式碼範例前先交代業務情境 |
| C4 | 推理路徑完整 | 長段落含觀察→詮釋→策略→執行 |
| C5 | 可查詢性 | 標題與關鍵詞 grep 友善 |
| C6 | 正面表述 | 規範性描述採「應該 / 建議」語氣，除 §4.3 三類例外 |
| C7 | 來源核對 | 引用的專案 / API / 統計數據皆有來源對照 |

### 5.2 負向表述專項掃描

在 C6 項目中執行以下關鍵字掃描：

```
掃描目標詞：不是 / 不可以 / 不可能 / 禁止 / 避免 / 不要 / 不得 / 錯誤的
```

對每個命中點執行下列判斷：

1. 命中點是否屬於 §4.3 三類例外（物理限制 / 安全警示 / 反模式對照）？
   - 屬於例外 → 保留，並在段落內說明歸屬類別。
   - 不屬於例外 → 依 §4.1 替換表改寫為正面表述。

2. 改寫後重讀該段，確認語意完整且建設性更強。

### 5.3 循環未通過時的處理

任一檢查項未通過時，**先改寫再繼續**。修改完成後重新執行 C1–C7 全項檢查，直到全部通過。

### 5.4 檢查結果記錄

每篇文章完稿後在檔案尾端加入一段檢查記錄（Hugo frontmatter 之外，以 HTML 註解保留）：

```html
<!--
codex-check:
  C1-intent: pass
  C2-atomic: pass
  C3-business-first: pass
  C4-reasoning-path: pass
  C5-searchable: pass
  C6-positive-framing: pass (0 negative-expression matches outside §4.3 exceptions)
  C7-source-verified: pass
  reviewed-at: YYYY-MM-DD
-->
```

此記錄作為後續審查的軌跡，協助辨識尚未通過檢查的草稿。

## 6. 工作流程

每次接到撰寫任務時依序執行：

1. **讀取權威 SKILL**：`compositional-writing/SKILL.md` 與（若涉及分析決策）`wrap-decision/SKILL.md`。
2. **依情境讀取 reference**：依 §2.4 觸發路由，只讀一份。
3. **錨定任務**：寫下本篇文章的目標讀者、要解決的問題、能獨立帶走的結論。
4. **起草**：依原子化、意圖顯性、業務邏輯前置原則逐段產出。
5. **逐段檢查**：每段完成後執行 §5 檢查循環。
6. **收尾記錄**：在文章尾端填寫檢查記錄（§5.4）。
