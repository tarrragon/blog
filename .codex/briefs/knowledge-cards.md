## 2. 掃描流程

### 2.1 術語提取（Divergent）

執行掃描指令，產出候選術語清單：

```bash
# backend 正文
rg -o '\[([^\]]+)\]\(\.\./[^)]+\)' content/backend/ --no-filename | sort -u

# go 正文中可能提及的後端術語（過濾英文大寫詞與常見技術詞）
rg -o '\b[A-Z][a-zA-Z-]{3,}\b' content/go/ content/go-advanced/ --no-filename | sort | uniq -c | sort -rn | head -100

# 中文書寫的技術術語（手動複核）
rg '(?:快取|佇列|訊息|事件|重試|降級|限流|熔斷|冪等|讀取模型|寫入模型|同步|非同步)' content/backend/ content/go/ -l
```

### 2.2 既有卡片比對

```bash
ls content/backend/knowledge-cards/ | sed 's/\.md$//' > /tmp/existing-cards.txt
# 將步驟 2.1 清單與 existing-cards.txt 比對，產出缺口清單
```

### 2.3 缺口歸類

每個缺口術語歸類到 `_index.md` 現有的主題分類之一：

- 資料與一致性 / 快取與流量 / 訊息與事件 / 可靠性與流量控制 / 可觀測性 / 部署與發布 / 安全與合規 / 事故應對 / ...

若某術語跨多個主題，優先選擇它**最常被使用的情境**所在分類。

### 2.4 優先級排序

| 優先級 | 條件 |
|-------|------|
| P0 | 被 ≥ 3 篇正文引用，且尚未建卡片 |
| P1 | 被 ≥ 2 篇正文引用 |
| P2 | 僅被 1 篇正文引用，但屬於 `_index.md` 現有主題分類的結構性術語 |

## 3. 卡片撰寫規範

### 3.1 檔案結構（依樣本提煉）

**正面範例標竿**：`content/backend/knowledge-cards/backpressure.md` 是本次任務的目標品質標竿。撰寫新卡片或補強既有卡片前請先閱讀它，觀察以下特徵：

- 定義段落把「要解決的問題」說清楚，並點出本概念的覆蓋範圍
- 概念位置段落除了列出現場，還說明與最常被混淆的相鄰概念（rate limit）的差別
- 可觀察訊號與例子分開寫，避免兩者互相擠壓
- 設計責任段落含有明確取捨說明（buffer 太小 / 太大的代價），而非單純清單
- 關鍵詞與相鄰卡片連結密度高，但不越界解說別張卡片的內容

**既有卡片的補強判定**：若某張既有卡片未達以上特徵，視為「薄卡片」，在 Task 1 掃描過程中一併列入補強清單，以與新卡片相同的品質標準補完。

**檔案結構**：

```markdown
---
title: "Term Name"
date: YYYY-MM-DD
description: "一句話說明這個術語要解決的問題"
weight: <依主題分類排序>
---

[術語] 的核心概念是「一句話定義，含關鍵角色與目的」。[補充 2-3 句，點出它在後端系統中的位置與取捨]。

## 概念位置

[說明它出現在哪些元件 / 流程中，與相鄰概念的關係。連結相鄰卡片用 `../xxx/`。]

## 可觀察訊號

[系統需要這個機制的訊號 / 遇到這個概念時會看到的指標 / 何時該考慮它。]

## 接近真實網路服務的例子

[用一個具體場景（電商、訊息通知、搜尋、訂閱）說明這個概念如何運作。避免把範例限縮到單一專案。]

## 設計責任

[導入這個概念後，團隊需要定義哪些邊界 / 決策 / 觀測項。]
```

**四段結構並非全部必填**：若術語較簡短（< 600 字），允許合併「可觀察訊號」與「例子」。以既有卡片（如 `cache-hit-miss.md` vs `broker.md`）作為體量參考。

### 3.2 連結策略

| 連結類型 | 做法 |
|---------|------|
| 卡片互連 | 術語首次出現時加 `[Term](../term-slug/)` |
| 正文 → 卡片 | backend / go 正文術語首次出現時加連結；同篇文章重複出現保持純文字 |
| 卡片 → 正文 | 卡片本體以保持原子化為主，避免指回正文 |
| _index.md 更新 | 新卡片補到對應主題表格，依 weight 排序 |

### 3.3 正面表述（對齊 `codex.md §4`）

- 以「應以 X 為準 / 優先採用 X / 搭配 Y 使用」取代「不要 / 不可 / 禁止」
- 保留否定詞僅限 §4.3 三類：**物理限制、安全警示、反模式對照**
- 範例：
  - 原句候選：「不要在 hot key 上用普通 cache-aside」
  - 正面改寫：「hot key 情境優先採用 singleflight 或 request coalescing，以避免快取同時 miss 造成下游壓力」（此處「避免」屬安全警示，保留並隱含理由）


### 3.4 反模式禁寫清單（對話式 AI 產出常見問題）

卡片屬於**參考型技術文件**，與 ChatGPT 式對話回應的敘事模式區隔明確。下表列出反模式特徵與對應的正確寫法，撰寫時主動規避：

| 反模式 | 具體特徵 | 正確寫法 |
|-------|---------|---------|
| Emoji 堆疊 | 標題 / 段首用 🧠 🔧 🧱 🔄 ⚖️ 🧵 💥 等裝飾 emoji | 純文字標題，使用 Markdown `##` 層級 |
| 口語對話開場 | 「你問...」「先說結論 👉」「我用實戰方式講」 | 陳述句開場：「X 的核心概念是...」 |
| 多概念混雜 | 一篇同時講 backpressure + rate limit + queue + pull-based + adaptive + concurrency + drop | 一張卡片聚焦**一個**概念；相鄰概念用連結引用 |
| 列舉式教學 | 「1️⃣ 最直接：限制輸入」「2️⃣ 用 Queue 吃掉尖峰」「3️⃣ Pull-based」...（大量平行列舉） | 四段結構（概念位置 / 可觀察訊號 / 例子 / 設計責任），避免把卡片寫成教科書章節 |
| 元對話收尾 | 「如果你想更實用一點，可以跟我說你在哪種場景」「下一則可以直接給你 Java 範本」 | 卡片結尾停在「設計責任」或 HTML 檢查註解，不延伸對話 |
| 強調語氣疊加 | 粗體 + 斜體 + 驚嘆號 + 👉 指向符號密集混用 | 關鍵術語用粗體或連結，指向符號僅限表格對照 |
| 混合抽象層級 | 同一段從「物理比喻（餐廳廚房）」跳到「程式碼偽碼」再跳回「控制理論」 | 一張卡片鎖定一個抽象層級；跨層需要時拆成相鄰卡片 |
| 過度鋪陳前言 | 「這就從概念進到工程實作核心了」「所有 X，本質都在調這三個旋鈕」 | 第一段直接定義，刪除鋪陳 |

**參考反例處理原則**：Codex 若收到類似「限制併發的手段跟控制詳細解釋」長篇 AI 產出作為素材，**提煉概念後拆成多張原子卡片**，每張聚焦一點，避免直接把長篇複製貼上。

**示例拆分方向**（僅為示範，實際拆法依概念歸類再定）：

| 長篇提及的概念 | 可能的獨立卡片 | 歸屬層級（參考 Task 2 brief） |
|--------------|--------------|--------------------------|
| 同時處理多少工作 | `concurrency-limit` | L2 機制層 |
| Semaphore 具體機制 | 評估：獨立卡片 or 寫入 go-advanced 章節 | 視跨域復用程度而定 |
| Worker pool 固定併發 | `worker-pool`（已存在） | L2 機制層 |
| Bounded queue + 拒絕 | `bounded-queue` | L2 機制層 |
| Pause/Resume 拉取節奏 | `pause-resume` | L2 機制層 |
| In-flight 追蹤 | `in-flight-count` | L2 機制層 |

實際拆分時，Codex 依「是否跨多個語境重複出現」「是否有獨立的可觀察訊號 / 設計責任」判定，而非機械按表格拆。

## 4. 執行步驟

1. **熱身**：讀完 §0 前置閱讀，熟悉 3 張樣本卡片。
2. **掃描**：執行 §2.1 三條指令，產出候選術語清單。
3. **比對**：對照 `existing-cards.txt`，產出缺口清單。
4. **分類 + 排序**：依 §2.3 歸類、§2.4 排優先級。
5. **批次撰寫**（每 5 張為一批）：
   - a. 撰寫 5 張卡片
   - b. 對每張執行 `codex.md §5` C1–C7 檢查循環
   - c. 執行 §5.2 負向表述掃描，有命中即改寫
   - d. 在每張卡片尾端加入檢查記錄 HTML 註解
6. **連結回填**：
   - 掃描 backend + go 正文，將新卡片對應的術語首次出現處加上連結
   - 更新 `_index.md` 主題表格
7. **完成回覆**：在對話中簡述本次新增卡片清單與回填連結的檔案，讓用戶確認結果。

| 編號 | 檢查項 | 通過條件 |
